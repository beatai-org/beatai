# 在 Netflix 让机器学习人人可用：构建模型生命周期图谱

[Saish Sali](https://www.linkedin.com/in/saishsali/)、[Nipun Kumar](https://www.linkedin.com/in/nipunk/)、[Sura Elamurugu](https://www.linkedin.com/in/suraelamurugu/)

## 引言

随着 Netflix 不断成长，机器学习持续支撑着 Netflix 向会员交付价值、在多个业务领域追求卓越。十多年前 Netflix 刚开始投入机器学习时，主要聚焦单一领域：个性化推荐。当时 Scala 是行业标准，ML 团队规模相对较小，优化会员参与度是主要用例。快进到今天，机器学习已成为 Netflix 业务转型的支柱。如今 ML 已应用到各个业务领域，包括：

- **个性化（Personalization）**：优化参与度，帮助会员发现喜爱的内容
- **工作室（Studio）**：制作前期与后期的工作流
- **支付（Payments）**：欺诈检测、支付路由、周期性账单优化
- **广告（Ads）**：最新的业务领域，需要实时决策与定向

…… 以及公司内部数量不断增长的更多用例

每个领域运行在不同的技术栈上，有着不同的业务指标和各自独特的组织结构。这种多样性证明机器学习已经演进到能在 Netflix 众多垂直业务中创造价值；但这种扩张也带来一个新挑战：让模型与数据在各领域之间交叉传播。

## 挑战：碎片化的 ML 版图

随着对 ML 的投入在这些领域不断扩张，一个关键问题浮现出来：产出的模型大多变成了黑盒。缺少发现基础设施，ML 从业者就难以跨业务垂直领域协作、共享成果。

来看一个具体例子：[内容嵌入（content embeddings）](https://netflixtechblog.com/mediafm-the-multimodal-ai-foundation-for-media-understanding-at-netflix-e8c28df82e2d)。Studio 团队打造了精巧的嵌入，用来识别场景边界、检测视觉转场、理解内容结构。这些嵌入最初是为制作工作流构建的。

但同一批嵌入在别处可能极具价值。广告业务设想中可以用内容嵌入做上下文匹配（确保广告与当前播放内容的基调、内容相契合）；个性化业务可以借助嵌入做分集化的商品营销与推荐（把某一集的主题或情绪与用户的观看偏好相匹配）。然而要让这种交叉传播真正发生却极其困难。

为什么？ML 工具各自困在孤岛里，每个都有自己的后端服务和用户界面。模型注册表（model registry）不知道有哪些 A/B 测试在用其模型，流水线编排器（pipeline orchestrator）也不知道下游有哪些模型依赖。ML 从业者要回答关于自身工作的基本问题，就得在多个系统之间来回穿梭：找一个模型要打开模型注册表，理清血缘要切换到流水线编排器，追踪哪些 A/B 测试在用该模型又得跳到实验平台。这种碎片化让从业者无法回答一些关键问题：

- **发现：** 存在哪些特征？有哪些数据源可用来为模型生成特征？
- **血缘：** 是哪条流水线在为某个特定模型生成数据？哪些数据源馈入了这些特征？
- **影响：** 哪些 A/B 测试在运行这个模型？改动某个特征会让哪些模型崩掉？这条链路上每一环的负责人是谁？

## 难题：把一切连接起来

真正的挑战不只是构建一个统一的 UI，而是要把 ML 从业者在生命周期各环节用到的那些零散基础设施连接起来。

Netflix 的 ML 生态系统从数十个来源产生元数据：

- 流水线编排系统发出执行细节、阶段依赖与数据转换
- 已部署的模型注册表追踪模型版本、产物、陈旧度与部署历史
- 实验平台管理 A/B 测试及其配置
- 特征存储（feature store）对特征定义和使用情况进行编目
- AI 数据集平台追踪数据集的创建、管理、发现与加载
- 身份平台维护用户、团队和组织的元数据

每个系统采用不同的格式、标识符和心智模型。必须解决的硬核技术难题是：如何收集这些异构元数据，将其转换成统一的实体模型，并构建一个连通的图谱，从而实现跨业务领域的真正探索与协作？

### 解决方案：元数据服务与模型生命周期图谱

答案是元数据服务（Metadata Service，MDS）——它构建出一个模型生命周期图谱（Model Lifecycle Graph），对 Netflix 全域的 ML 相关实体进行索引和连接。MDS 针对 ML 元数据（模型、特征、流水线、实验、数据集等）的实时摄取做了优化，用来回答"哪些实验在运行这个模型？""哪些模型共享这些特征？"之类的跨领域问题。MDS 是支撑发现能力的基础：从多样来源摄取事件，结合上下文加以丰富，再把实体之间的关系物化出来。

我们的愿景是：让 Netflix 的每一项 ML 资产，都能被每一位 ML 从业者发现、理解和复用，无论其身处哪个团队或领域。

## 核心抽象：系统的词汇表

在深入技术实现之前，先理解支撑 MDS 的概念模型会有帮助。这套词汇让团队与系统之间保持一致的沟通：

**组件（Component）：** 任何能用 AI 平台（AI Platform，AIP）的统一资源标识符（URI）唯一寻址的对象。AIP URI 遵循 `aip://<componentType>/<platformId>/<resourceId>` 的格式，确保全局唯一。例如：

- 模型：`aip://model/registry/ranking-v5`
- 用户：`aip://user/identity/alice`
- 流水线：`aip://pipeline/orchestrator/weekly-training`

**实体（Entity）：** ML 生态系统中的一个组件，由名称、描述、创建日期、负责人等额外属性来刻画。实体代表 ML 专属的资产，比如模型、特征和流水线。

**实体类型（Entity Type）：** 一组共享相同数据形态（data shape）的实体。数据形态是一组属性约束，规定一个实体必须具备哪些属性和关系。

**领域（Domain）：** 一组相关实体类型的功能性分组，为某一类 ML 资产定义抽象接口。例如，Models 领域定义了 Model 和 Model Instance 的样子，Pipelines 领域则定义 Schedule、Request 和 Execution。

**提供方（Provider）：** 某个领域的具体实现，由特定来源系统支撑。例如，Models 领域目前由 Netflix 内部的模型注册表支撑。这种分离让 MDS 能为同一领域支持多个提供方——若引入一个新的模型注册表，可作为额外的提供方接入，无需改动领域接口。

可以用一个具体例子来概括这些概念：

![](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@d636560ddb58a0d75173d1977cf7a323f1319997/ai-insights/2026-05/22/images/democratizing-machine-learning-at-netflix-building-the-model-lifecycle-graph/01.webp)

这种基于 URI 的寻址方案至关重要——让任何服务都能用一个字符串引用任何 ML 资产，而 MDS 能把该引用解析回丰富、连通的元数据。

## 从事件到实体再到图谱

从原始系统事件走到一个可查询的图谱，是分阶段发生的。下面用一个具体例子走完每个阶段：通过关系推断，把模型连到与之关联的 A/B 测试。

### 1 事件摄取

MDS 通过 Kafka 和 AWS SNS/SQS 与各类来源系统集成，实时消费事件。来源系统只发出"轻量"事件，仅包含一个标识符和一个事件类型。

事件示例：

```json
{
  "event_type": "model_instance_created",
  "instance_id": "ranking-model-v5-20XX0101",
  ...
}
```

这种设计让生产方保持简单：来源系统只需宣告发生了一次变更，无需构建完整载荷，也无需理解下游需求。

每个来源系统在 MDS 里都有专属的事件处理器：

- **流水线编排（Pipeline Orchestration）**：摄取流水线执行事件，包括节点定义、调度、请求和作业尝试
- **模型注册表（Model Registry）**：捕获模型部署、配置和版本更新
- **特征存储（Feature Store）**：追踪特征定义及其版本
- **实验平台（Experimentation Platform）**：监控 A/B 测试的配置与分流
- **数据集（Datasets）**：追踪 ML 数据集及其版本
- **身份平台（Identity Platform）**：维护所有权和团队成员信息

### 2 实体丰富

MDS 为每种事件类型实现了一份注水契约（hydration contract）。事件到达时，MDS 会：

1. 校验事件的 schema
2. 调用来源系统的 API，获取完整的当前状态
3. 把响应转换成一个归一化的实体

这种设计有一个关键特性：事件顺序无关紧要。MDS 总是从事实源（source of truth）获取最新事实。这个模式把事件流与状态一致性解耦——即便事件总线丢了一条消息、或乱序投递，下一个事件也会修正状态。于是事件流成了一份变更通知，而非一份变更日志。

这种"变更通知"模式带来了几个重要权衡。好处是：让生产方保持简单，对乱序或丢失的事件足够健壮，并确保 MDS 总能通过读取事实源调和到最新状态。代价是：注水期间会给来源系统增加额外的读取负载，因此丰富工作进程必须刻意做好限流、缓存和退避，以免把来源系统压垮。

回到排序模型那个例子：当 `model_instance_created` 事件到达时，MDS 会调用模型注册表的 API：`GET /api/v1/instances/ranking-model-v5-20XX0101`

注册表返回一个完整的描述符。响应示例（仅含关键字段）：

```json
{
  "id": "ranking-model-v5-20XX0101",
  "pipeline_run_id": "train-weekly-ranking-20XX0101",
  "owner_emails": ["alice@netflix.com"],
  "labels": [{"key": "team", "value": "personalization"}],
  ...
}
```

### 3 数据转换与归一化

原始事件是异构的，每个来源系统都有自己的 schema 和语义。MDS 的工作进程把这些事件转换成一个带有标准化字段的统一实体模型。

如果不做归一化，下游消费者就得理解每个来源系统的 schema。归一化创造了一个一致的接口，让查询和关系能够跨所有实体类型工作。下面是一个例子。

归一化后的 MDS 实体：

```json
{
  "id": "aip://model/registry/ranking-model-v5-20XX0101",
  "pipeline_run": "aip://pipeline-run/orchestrator/train-weekly-ranking-20XX0101",
  "entity_type": "ModelInstance",
  "owners": ["aip://user/identity/alice"],
  "tags": [{"tag": "team", "value": "personalization"}],
  ...
}
```

归一化过程会标准化字段名和格式。例如，平台专属的 ID 变成全局 AIP URI，`owner_emails` 变成带有已解析用户 URI 的 `owners`，`labels` 变成 `tags`；`pipeline_run_id` 之类的外键被转换成实体引用。不过，此时仍然没有任何引用指向"哪些 A/B 测试在用这个模型"。模型注册表不追踪实验，实验平台也不追踪是哪条流水线产出了某个模型——这正是知识丰富变得至关重要之处。

### 4 存储与索引

一旦归一化完成，实体就被持久化到 Datomic，并立即在 Elasticsearch 中建立索引。这是在事件处理流程内同步发生的。

**用 Datomic 做缓存和关系**  
归一化后的实体先写入 Datomic；Datomic 既是本地缓存，也是图数据库。

为什么选 Datomic？它既是 MDS 的记录系统（system of record），也是丰富过程所用的工作数据集。其不可变的事实模型意味着可以持续添加关系，而不丢失原始的实体状态。

**存储的内容：**

- 全部实体属性，以事实形式存储
- 实体引用（外键，可能指向尚未完全解析的实体）
- 全部关系，以具象化的边的形式存储（由丰富过程添加）
- 实体的生命周期状态（追踪哪些实体已完全丰富、哪些还在等待注水）

**由此带来：**

- **复杂的图遍历：** 一次查询就能从一个模型导航到其特征、再到这些特征的数据源
- **实体关系：** 跨多个领域做连接（join），不会遇到 N+1 查询问题
- **灵活的 schema 演进：** 随着目录增长，轻松添加新的实体类型和属性
- **渐进式丰富**：后台作业高效识别并处理需要额外注水的实体，让图谱逐步完善，而无需重新处理已完全丰富的实体

实践中，Datomic 用于关系密集型的导航式查询，例如：

- 从这个模型实例出发，列出所有上游数据集和下游实验。
- 给定这个特征，列出所有消费它的模型，以及拥有这些模型的团队。

这些查询常常跨越图谱中的多个跳数，能受益于 Datomic 不可变的事实模型，以及跨实体关系的高效连接。

**用 Elasticsearch 做发现**  
写入 Datomic 之后，实体立即在 Elasticsearch 中建立索引，以支撑跨目录的快速全文搜索。

**索引的内容：**

- 主要字段：实体名称、描述、实体类型、负责人姓名
- 关系元数据：相关实体的名称（例如某个模型的特征、流水线、A/B 测试），存储在 `related` 字段中
- 标签：领域专属的元数据，以键值对形式存储（例如 `team::personalization`、`env::production`、`model.state::released`）

**索引结构：**

- 单一的 entities 索引：所有实体类型（模型、特征、流水线等）索引进一个统一索引，靠 `entityType` 字段区分
- 独立的 owners 索引：为用户和群组设的专属索引，支持跨实体的负责人搜索
- 相关性加权：精确名称匹配的得分高于其他相关匹配

**由此带来：**

- 跨实体名称、描述、标签和相关元数据的多字段文本搜索
- 带加权的相关性排序（精确名称匹配得分显著更高）
- 按实体类型、所有权、标签和领域专属属性（以标签形式存储）进行复杂筛选
- 模糊匹配，以应对拼写错误和不完整的查询

Elasticsearch 支撑着进入系统的入口：用户通常先在 AIP 门户里做一次自由文本搜索（搜模型名、团队或领域术语），落到某个实体页面后再切换到图导航。索引作为摄取和丰富工作流的一部分，几乎实时建立，所以变更通常只带很短的延迟就显示在门户里——这个延迟对交互式使用完全可以接受。

### 5 知识丰富与图谱形成

一旦实体元数据被持久化到 Datomic，定时的后台进程就接手去发现并物化关系。这些丰富作业周期性运行，扫描那些未缓存或仅部分解析的实体（即仅以引用形式存在、没有完整元数据的实体）。

丰富工作流：

- **识别候选：** 找出被标记为未缓存、或带有未解析引用的实体
- **注水关系：** 查询事实源系统，获取相关实体的细节
- **物化边：** 把发现的关系写回 Datomic
- **重新索引：** 为更新过的实体触发 Elasticsearch 索引
- **标记为已丰富：** 更新实体状态，避免冗余的重复处理

这种异步方式让 MDS 能承担图谱形成的计算成本，而不阻塞实时的事件摄取，同时还支持重试逻辑、以及随新实体出现而进行的渐进式丰富。

由于丰富是异步的，新发现的关系可能在底层实体创建后带一个很短的延迟才出现（通常是分钟级而非秒级）。MDS 会追踪每个实体上一次被丰富的时间，并在 AIP 门户里呈现这个时间戳，这样从业者就能判断陈旧度，知道何时可以放心依赖某个关系来做调试或影响分析。

**为什么要丰富？** 来源系统是为特定目的而构建的，并不了解其他领域里的实体。丰富会发现并物化跨系统的关系，从而支撑强大的血缘和影响查询。

### 示例：把模型连接到 A/B 测试

MDS 处理一个新的模型实例时，后台丰富作业通过多跳推断来发现关系：

**步骤 1：直接连到流水线**

模型引用了一个 `pipeline_run_id`。一个丰富作业对流水线注水，发现与之关联的 A/B 测试：`GET /api/v1/pipeline-runs/train-weekly-ranking-20XX0101`

响应：

```json
{
"run_id": "train-weekly-ranking-20XX0101", "pipeline":  "weekly-ranking-trainer",
"ab_test_cells": [
   {"test_id": "12345","cell_number": 2,"cell_name": "treatment_ranking_v5"}
 ]
 ...
}
```

**步骤 2：发现 A/B 测试的上下文**

丰富作业发现这条流水线是为 A/B 测试单元 #2 运行的，于是向实验平台查询测试细节：`GET /api/v1/tests/12345`

```json
{
 "test_id": "12345",
 "name": "Ranking Model v5 vs v4",
 "status": "ACTIVE",
 "cells": [{"cell_number": 1, "name": "control_ranking_v4"}],
 ...
}
```

**步骤 3：推断传递性关系**

丰富作业现在握有了完整的链条：

- 模型实例由某次流水线运行产出
- 该次流水线运行是为 A/B 测试单元 #2 执行的
- A/B 测试单元 #2 属于 A/B 测试 "Ranking Model v5 vs v4"
- 模型实例现在被关联到这个 A/B 测试

该作业把推断出的关系写回 Datomic，触发重新索引，并在图谱中物化这些边。MDS 不只是存储外部告知的信息——还会通过在后台遍历图谱，推导出新的知识。

**这为什么重要：** 没有 MDS 时，要回答"哪些 A/B 测试在用这个模型？"需要：

1. 在模型注册表里查这个模型
2. 查清是哪条流水线产出的
3. 在流水线编排器里检查 A/B 测试标签
4. 向实验平台查询测试细节

有了模型生命周期图谱，这只需一次查询：

```
query {
  model(id: "aip://model/registry/ranking-model-v5-20XX0101") {
    name
    owners { name }
    currentInstance {
      version
      pipeline {
        name
        owners { name }
      }
      features {
        edges {
          node {
            name
            data { edges { node { name } } }
          }
        }
      }
      associatedAbTests {
        name
        cells { number name }
      }
    }
  }
}
```

反向查询同样可行："实验 12345 里正在测试哪些模型？"

## 实现探索，而不只是搜索

![](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@d636560ddb58a0d75173d1977cf7a323f1319997/ai-insights/2026-05/22/images/democratizing-machine-learning-at-netflix-building-the-model-lifecycle-graph/02.webp)

有了模型生命周期图谱，工作方式就从实体搜索转向了实体探索。发现不只是找到一个模型，更在于遍历各种关系：

- 从一个模型出发，探索其特征
- 从特征出发，导航到驱动这些特征的核心数据
- 从数据出发，回溯到生成数据的流水线
- 从流水线出发，看清流水线归哪些团队拥有、被哪些团队依赖
- 从实验出发，理解哪些模型正在被测试

举例来说，设想一位工程师正在排查某个个性化模型参与度指标的下滑，他可能会：

1. 在 AIP 门户里，从为受影响推荐供能的模型实例入手。
2. 检查该模型的特征，顺着一个可疑特征追到上游数据集。
3. 在数据集页面上，看到其流水线最近有过失败的运行，并锁定拥有该数据集的团队。
4. 确认当前有哪些 A/B 测试正在运行这个模型实例，从而了解哪些会员和界面受到了影响。

在 MDS 和模型生命周期图谱出现之前，这需要在多个工具（模型注册表、流水线编排器、实验平台）之间手动来回核对。如今，这一切在单一界面里变成了一段连贯的旅程。

这种基于图谱的探索能回答那些以前根本无法回答的问题：

- 血缘查询：这个模型的完整血缘是什么，从训练数据一直到生产环境的实验？
- 影响分析：改动这个特征会波及哪些模型？
- 使用发现：哪些 A/B 测试在用这个模型？
- 依赖映射：某条流水线在传递关系上依赖哪些数据源？
- 弃用规划：哪些实体已不再被使用、可以退役了？

每个实体都有深厚的上下文：创建时间、所有权、更新历史，以及最重要的——与其他实体的关系。

模型生命周期图谱通过 AIP 门户呈现给从业者——这是一个统一的界面，提供跨所有实体类型的全文搜索、带有可导航关系的详细实体页面，以及为团队和个人定制的个性化视图。

在 AIP 门户里一次典型的交互是这样的：

- **搜索：** 在那个由 Elasticsearch 支撑的搜索框里，输入模型、特征、数据集或团队的名称。
- **检查：** 落到一个实体页面，上面展示关键元数据（描述、负责人、领域、标签），旁边还有一个关系面板。
- **探索：** 点进相关实体（上游数据集、下游实验、同辈的模型版本），不离开门户就能导航模型生命周期图谱。

新的实体类型被引入 MDS 时，门户会自动提供基线的搜索、实体页面和关系导航，随后可以再逐步叠加领域专属的可视化（比如模型部署历史、数据集版本时间线）。

## 前路：尚未解决的挑战

构建 ML 生命周期图谱是一段持续的旅程。重大挑战依然存在——这些挑战也正是未来的机会所在：

- **工具激增：** 新的 ML 工具不断涌现，需要能规模化的健壮集成模式。如何设计插件式架构，让接入新来源变得无缝？一旦跟不上新工具，从业者就会被迫退回碎片化的视图，模型生命周期图谱也会失去覆盖度与信任。
- **领域专属的可视化：** 不同实体类型需要各自不同的可视化体验。模型页面应展示部署历史、A/B 测试关联和性能指标；特征页面应突出数据血缘和消费它的模型；流水线页面必须展示执行历史、依赖关系和调度；数据集页面需要版本时间线和下游消费者。如何设计一个灵活的 UI 框架，让每种实体类型都拥有量身定制的体验，同时在整个门户里保持一致的导航与交互模式？缺少丰富的、领域专属的体验，门户就有沦为通用目录的风险，而非 ML 从业者日常工作流中真正依赖的工具。
- **元数据质量：** 如今，MDS 通过事实源注水和摄取时的 schema 校验确保数据一致性，后台丰富作业则持续推断关系、从来源系统物化实体。然而在规模化场景下确保完整性与及时性，挑战依然存在：来源系统未能发出事件、所有权信息变陈旧、实体缺乏描述与上下文元数据时，图谱的效用都会退化。如何构建自动化的校验与丰富系统，来检测元数据异常、建议缺失的关系、在数以百万计的实体上维持质量基准？糟糕或陈旧的元数据会侵蚀从业者的信任——图谱一旦不完整或不正确，团队就会退回到临时知识和一次性集成，不再把 MDS 当作事实源。
- **高级关系推断：** 在来源系统声明的显式关系之外，如何推断出隐式的连接？能否基于共享的特征，判断出两个模型服务于相似目的？能否基于相似流水线的使用模式来推荐特征？这些想法的探索还处于早期阶段。一旦做好，就能把 MDS 从一个被动目录变成面向 ML 资产的主动推荐引擎，加速复用，减少各领域间的重复工作。
