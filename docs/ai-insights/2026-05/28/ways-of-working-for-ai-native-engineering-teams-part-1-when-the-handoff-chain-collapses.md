# AI 原生工程团队的协作方式，第一部分：当交接链条崩塌时

在我作为工程负责人和顾问的大部分职业生涯里，一直围绕一条交接链条来组织工程团队：PM 把 PRD 交给 UX 设计师，UX 设计师把视觉稿交给工程师，工程师把构建产物交给 QA。在 AI 原生团队里，这条链条正在崩塌——而维系它的角色边界，也正随之溶解。

这是一个系列的**第一部分**，讨论当 AI 被当作软件构建、测试、部署、文档、维护的核心运行层——而非外挂上去的某个功能时——***协作方式***究竟会变成什么样。

当一个团队走向 AI 原生的软件开发时，这六块里每一块都得变。这篇文章只啃第一块、也是最根基的一块——**角色与职责**——因为其他每一块都建在它之上。后面第二到第六部分会逐一覆盖其余几块。

## **我在 Terraworks AI 看到的**

[Terraworks AI](https://terraworks.ai/) 是我联合创办的一家教育科技初创公司，目标是改善新兴市场学习者的学习成果——也是我观察 AI 原生模式发展最贴近的地方。在那里真正让我意外的，不是某个 agent 或某项 skill，而是*团队把 agent 带进会议室的方式*。

当一个新功能或重要功能立项时，PM、设计师、工程师/技术负责人会一起围攻这件事。PM 会开一个 agent 会话，触发一个*需求收集* skill，澄清*为什么*要做、收敛范围、对假设做压力测试。其他角色不会在旁边干等——会一起进会话、跟着出力、提问、把上下文喂给 agent。两三个小时之内，agent 和三个人就在一个共享的设计概念上对齐了。PRD（Product Requirements Document）只是产出物，真正的输出是对齐本身。

设计阶段也是一样的跑法。团队会刻意分头——每个成员各自跟一个 AI 设计 agent 独立勾画一个不同方向。（这种"先发散再收敛"的模式借鉴自 Jake Knapp 的 [*Sprint*](https://a.co/d/019YRbVd)。）然后再回到一起，用 AI 把这些想法合并，由设计师运用自己的领域专长敲定最终版本。

工程接力从这里开始：PRD 加上设计产物，加上既有代码库，喂给一个 AI Agent，定义技术需求，产出技术设计文档（TDD）和 ADR（Architecture Decision Records）。

而在每一次这样的会话里，人都还在背景里做一件事：留意 agent 哪里不够用——缺哪个 skill、prompt 不够好、context 有缺口——并当场补上。团队是在交付的同时，自己给自己加杠杆。

这不是工作流上的微调，而是另一种运行模式。链条不只是变快了，而是塌缩成了一个闭环。

## **核心论点：角色边界正在溶解**

我看到的现象底下藏着一个模式：

> 在 AI 原生团队里，角色之间的边界正在溶解。原本以交接链条流动的工作（PM → 设计 → 工程 → QA）现在以协作闭环的方式流动：人在上游共同塑造规范，agent 在中段做实现，人在下游做验证和评审。

角色本身没有消失，而是围绕一个新的重心被重新分配。有价值的人类工作——那些*不可验证*的工作，需要判断力、"完成"的定义又模糊——会向闭环的两端聚集。中段那些可验证的工作，越来越多地交给 agent 来跑。

如果你带一支工程团队，那职责就是围绕这个闭环重新设计团队结构和实践——而不是去保住那条旧链条。

接下来按工作流动的顺序，逐个角色说说这种再分配长什么样。

## **PM/PO：从规范作者，变成围攻协调者**

旧模式里，PM 是 PRD 的责任人。他们对接利益相关方和客户，写下需求，然后交出去。

在 AI 原生团队里，PM 仍然是*为什么*的责任人——但*做什么*越来越多地是在一场围攻里共同写出来的。PM 最高杠杆的时间，不再花在 Notion 里敲字，而是花在跟一个 agent、加上团队其他成员一起开一场需求收集会。他们成了 agent 与跨职能小组之间一场结构化对话的协调者。

**你作为负责人应该做什么：**

-   招聘并培养能在持怀疑态度的工程师面前，跟 agent 进行现场对话的 PM，而不只是能产出文档的 PM。
-   投资于共享的、有版本控制的 agent skill，比如*需求收集*，让各团队的围攻有一致的起点。

## **设计师：从规范消费者，变成多方向生成者**

旧模式里，设计发生在 PRD 之后——往往要经过一轮独立的 UX 调研周期，配上原型、用户访谈、Figma 交付件。

在 AI 原生团队里，设计阶段被压缩成"发散—收敛"：每个团队成员跟一个 AI 设计 agent 去探索一个不同方向，然后整队合并。设计师的贡献从"设计的唯一作者"变成"领域专家"——他们对团队和 agent 共同生成的方向做*判断和精修*。

在这之下还有一个更激进的转变，我已经看到苗头，但还很早。**AI 原生团队正在压缩——有些团队甚至完全砍掉——前期那段做 UX 调研原型的阶段。**团队把想法直接上生产，让真实用户来检验。如果效果不好，就再发一版。我听过 Anthropic [谈过](https://youtu.be/igO8iyca2_g?si=6oBR08VQUYRlTgcn)他们内部的做法，预计未来 12 个月里会有更多团队跟进——但这一条是预测，不是我已经在各处都见到的情形。

**你作为负责人应该做什么：**

-   按工作的新形态招设计师：能跟 PM、工程师、AI Agent 在围攻里协作；能把多个 AI 生成的方向合并成一份连贯的设计；能用 taste 和判断力挑出正确方向——而不只是能从一张空白画布上原创一份精致设计。
-   有意识地判断你的领域是否属于那种"前期原型与调研阶段*不应*被压缩"的领域（受监管？关乎安全？），并在该保留的地方守住它。

## **工程师：从一行一行的实现者，变成架构师和评审者**

旧模式里，工程师的手艺活在实现：写代码、组织模块、把测试跑绿。

在 AI 原生团队里，工程师的手艺活向*上游*的技术设计和*下游*的评审两端转移。他们跟一个 AI 架构 agent 配合，从 PRD 和既有代码出发，产出技术设计文档和 ADR。他们把工作切成垂直切片、可验证的工单——往往有一个 AI agent 帮忙切，由工程师把关切分质量。他们把这些工单丢给一个开发 agent——在自己机器上跑也好、作为云端 agent 跑也好——大半个实现过程他们都可以离开键盘。

当 agent 把提交回送过来时，工程师的主要工作才开始。他们要对那些 agent 自己暂时还无法验证的东西做评审：

-   [**技术债**](https://martinfowler.com/bliki/TechnicalDebt.html)——抽象放错了位置，今天看上去没事、半年后引爆的捷径。
-   **认知债**——能跑、但没人能在脑子里装下的代码（[Margaret Storey 最近写过](https://margaretstorey.com/blog/2026/02/09/cognitive-debt/)）。
-   **运维债**——能上线、但运行成本高、难以观测或难以恢复的代码（[Darren Mothersele 一直在 LinkedIn 上写](https://www.linkedin.com/posts/darren-mothersele_technical-debt-is-understood-were-starting-activity-7434908785699332096-5BMO)）。
-   **产品 taste**——能跑、但不像这个产品该有的样子：交互别扭、文案语气不对、默认值跟用户真正想要的不一致。

这里的纪律，就是 Claude Code 团队的经理 Fiona Fung 所说的[**信任但要验证**](https://youtu.be/igO8iyca2_g?t=880)。评审这一步也是你评估 agent 表现的环节——你要判断当时是什么 skill、什么工具、什么验证器本可以挡住那个缺口，从而让下一次会话比上一次更锋利。

**你作为负责人应该做什么：**

-   提拔并奖励那些在设计、拆解和评审上很强的工程师。他们现在是团队里杠杆最高的个人贡献者。
-   停止用代码行数或提交数量来衡量工程产出——这些指标衡量的是 agent，不是人。

## **QA：被并入工程师**

QA 工程师正在被并回软件工程师里。这一幕我在团队里直接看到过。

实事求是地说，这是一个早已存在的趋势的*加速*，不是一个全新趋势。很多小团队从来就没有专职 QA 工程师——软件工程师自己扮演这个角色。AI 原生工程之所以加速这种合并，是因为传统 QA 职能的大头（测试生成、回归覆盖、探索性测试骨架）正在被 AI QA agent 增强。留给人的，跟代码评审里留给人的那一层一样：不可验证的部分——*"这真的是我们想要的吗？这对用户真的能用吗？"*——再加上上游的工作：把边界情况清晰地写下来，让 agent 去追。

那段上游工作本身就是一项 skill。AI 原生团队里的软件工程师，现在需要在"思考边界情况"上格外擅长，并且能把边界情况表达得足够精确，让 agent 据此行动。

**你作为负责人应该做什么：**

-   如果团队仍然有独立的 QA 职能，主动地规划合并，而不是让它通过自然流失自己发生。决定现有 QA 工程师将成长成什么角色。
-   招那种把"边界情况枚举"当成一等技能、而不是当成苦差事的工程师。

## **混合体：AI 设计与开发工程师**

这一切的极限情形，是我已经在团队里看到的一个角色：**AI 设计与开发工程师**。一个角色戴上所有帽子——产品思考、设计感、软件开发——靠一支能力到位的 agent 队伍来重度增强。他们跟不同 agent 配合扮演不同角色，然后把东西交付出来。

在我观察过的团队里，这个角色被用在**内部工具和非核心功能**上——这类工作单个人的判断就够用，跑一个重量级的跨职能流程反而过头。我（目前还）不会把它放到受监管的支付流，或者百万用户每天看到的核心界面上。但这个角色本身存在的事实，已经告诉了你趋势指向哪里：随着角色边界溶解，极限情形是*一个人*带着 agent 撑起整个闭环。

**你作为负责人应该做什么：**

-   承认这个混合角色存在。给它一个职位坑，判断它适合放在哪里，别硬塞进现有的职级阶梯里。
-   也要诚实面对它当前还不适合的地方。混合角色是某些工作的工具，不是对高赌注工作中跨职能围攻的替代品。

## **EM / 技术负责人：从管人，变成管人和管 agent**

如果你读到这里，这一节谈的就是你的工作。

工程经理的角色，从管人，转变为同时管**人（他们本身又是 agent 队伍的管理者）和 AI agent**。具体来说：

-   **设计能最大化 AI 杠杆的团队结构和工作流。**这是这份工作的新核心。团队的吞吐量，现在主要是*工作如何被塑造*的函数，而不是你往上面堆了多少人。
-   **确保团队真的在把活下放给 agent。**在旧模式里练出手艺的工程师会本能地伸手去抓键盘。EM 工作的一部分，就是不断地把他们往上游推到设计和评审里，那是现在他们判断力最值钱的地方。
-   **管护 agent 的 skill 和能力。**当团队在某次会话里发现 agent 不够用时，经理的任务是确保这些缺口转化成投资：一项新的 skill、一个更锋利的验证器、一件更好的工具。团队的杠杆是在这一层复利累积的——而且只有这一层。

**你作为负责人应该做什么：**

-   把 1:1 和团队评审重新围绕新核心来组织：人有没有在下放？这周 agent 在哪些地方不够用？我们又往 agent 工具箱里加了什么？
-   明确切出团队时间做 skill 和工具的投资。不切的话，这事就不会发生——你本应复利累积起来的杠杆，会悄悄漏光。
