# 不要被动反应，要主动回应：我们为何重构了 SDLC 来守住专注

创业文化建立在多任务并行之上，可人脑不是这么造的。本文讲讲我们怎样重新设计 Ewake 的 SDLC，不惜代价守住团队的专注力。

![一张极简风格的传送带工程示意图。左侧，一只机械机器人靴子把一堆杂乱的干扰物——包括 Slack 图标、电子邮件信封，以及一个标着"嘿，就一件小事！"的盒子——从传送带上踢下去。靴子之后，传送带平稳向前，只承载着干净、间距均匀、泛着绿光的几何立方体，代表专注、不被打断的工作。](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@d862590ea9216ae63f9cd9b4163d944ec90098a3/ai-insights/2026-06/04/images/dont-react-respond-why-we-rebuilt-our-sdlc-for-focus/01.webp)

我是个反多任务的卢德分子，从来都不喜欢多任务并行。这是我们这行的职业病，而代码生成只让它变得更快（也更糟）。我一直觉得，没完没了地多任务并行是种糟糕的活法，如今这想法只增不减。

可悲的是，证据竟然来自一条 Instagram 短视频。不过容我辩解一句，这条短视频的引用之扎实，相当少见。我从刷得停不下来的负面信息流里抽身，去顺藤摸瓜查他们引的研究，一头扎进兔子洞，最后带着*证据*钻了出来。

事实证明，神经科学界的共识相当明确：多任务并行是个神话。人脑的构造根本不支持它。我们*以为*的多任务并行，不过是快速的任务切换罢了。

研究人员就此拉响警报已有一阵子了。[Mark 等人 (2008)](https://dl.acm.org/doi/10.1145/1357054.1357072) 证明，在任务间反复横跳或许短期内让人觉得更快，但终究会导致倦怠。哪怕让人完全自由地安排日程，那些选择多任务并行的人，表现也明显不如逼自己按顺序逐项做完的人 ([Buser & Peter, 2012](https://www.cambridge.org/core/journals/experimental-economics/article/multitasking/0D8CADEA4C6F6ACDBB1E51615B5D67DA))。

这种快速的任务切换不是免费的，甚至谈不上廉价。每次切换注意力，大脑都得卸下当前的认知框架，再装载一个新的。[Hasan (2024)](https://journals.lww.com/annals-of-medicine-and-surgery/fulltext/2024/11000/digital_multitasking_and_hyperactivity__unveiling.5.aspx) 发现，不断重新定位自己所需的心智负荷，会吃掉**高达 40%** 的有效工作时间。更糟的是，这样切分资源会让工作记忆超载，直接导致精神疲劳和决策变差。

既然多任务并行让我们更慢、把工作做得更差，还实打实地消耗心理健康，那我们为什么还乐此不疲？

因为多巴胺的错觉。神经科学家 Daniel J. Levitin 在他的著作 [*The Organized Mind*](https://www.daniellevitin.com/the-organized-mind) 里解释道，人脑天生就会奖励新鲜感。每次切换上下文，大脑都会因为挖到新信息而分泌一阵多巴胺。你觉得自己很高效，因为你确实在忙活"做事情"。可实际上，你只是在奖励大脑分心而已。

**tl;dr**：Cian 读了 3（三）篇研究论文和一本书的梗概，它们无一例外都支持他的观点——没完没了的多任务并行又慢又伤身。

40% 这个数字有点吓人，其余的则只是印证了我早有的判断。

当我带着多巴胺错觉这个视角去审视我们的工程流程时，发现它无处不在。我们那套"敏捷"的创业文化，其实不过是给全团队、制度化的多任务并行找个借口。

## 我们在原地打转

刚起步时，我们做着典型的 Startup™️ 那一套。直觉告诉我们，从想法到信号，最短的路就是对的路。我们每周规划一次，几乎每个周期都被客户需求打断。我们着手做某件事，一直做到冒出更重要的事为止——而这种事经常冒出来。

我清楚地记得九月份的某一周。我们整周都埋头于"真正"的工作，腾不出时间规划，周五走进规划会时，手里只有几个半成品想法。那次会议是我们头一回真正拷问这些想法，二十分钟内就看清它们还没成熟。于是我们只好周一再规划一遍。我们就是在原地空转。

关于这件事的 PR 队列那一面，我们[之前写过](https://www.ewake.ai/blog/the-cost-of-cheap-code)——当评审节奏跟不上新代码产出的速度时会发生什么。那是战术层面。这篇讲的是战略层面，它在任何代码动笔之前就开始了。单看个人，我们感觉很快。可摊到一个月，我们做成的事反而更少。这笔成本只是没记在账上。

## Shape Up 精简版

我们四处物色方法论，最后选中了 Shape Up，看中的是它处理范围的方式。与其估算一项工作要花多久（在代码生成工具的当下，这事既别扭又困难），我们改为预先定下一个"意愿额度"(appetite)。我们会说："作为一个团队，我们愿意在这个项目上投入这么多时间。"一旦触到额度上限，我们要么削减范围，要么提高额度。关键在于，我们就"接下来怎么做"明确做出决定。

我们用一份叫 one-pager（单页文档）的文件来预先设计范围。它明确写清意愿额度、范围内有什么、（最关键的）范围外有什么，以及哪些风险因素可能让项目超出额度。

每个项目都从一份 one-pager 起步。我们越来越擅长提前界定范围。但我们当时还没准备好全盘采用 Shape Up 的"活跃周期 + 冷却期"模式，于是仍然每两周照常做一次冲刺规划，和它并行。

我们做的另一项改动，是技术设计的方式。我们本想并行推进，结果搞成了各自为政。我们老是把设计讨论拖到 PR 阶段，等代码都写完了，才去争论底层方案对不对。于是我们开始强调：动代码之前，先在纸上把设计敲定。

## 把工作左移

道理很简单：在范围上多花时间，能省下更多设计时间；在设计上多花时间，又能省下更多代码评审和实现的时间。

效率提升立竿见影。原本要在 PR 上耗一整天的设计争论，挪到设计评审阶段一小时就搞定。遗漏的需求在范围评审时就被逮住，不必事后返工补救。糟糕的功能在落地之前就被砍掉，而不是让它的 PR 卡在那里。

代码生成让这件事更重要了。[Fadó fadó](https://en.wiktionary.org/wiki/fad%C3%B3_fad%C3%B3)（很久很久以前），工程师实现一个功能要花三天，我们还有时间边做边对设计做方向性调整。Claude Code 则是"发射后不管"。你要是把它指向错误的方向，它会眨眼间高高兴兴地给你实现出一个糟糕的设计。如今生成代码几乎是免费的，人类的架构思考反倒成了我们 SDLC 里最值钱的东西。动手之前就清楚知道自己要造什么，已经不是可选项，而是工作的全部。

有那么几周，这感觉像是顿悟。我们修好了范围界定的问题，PR 队列重新喘上了气，工程师们也觉得自己对技术选型有了真正的话语权。

于是一切就绪。问题解决了，对吧？

## 没那么简单。

我们仍然在"真正"的工作之间硬挤出时间写 one-pager。我们很少有空把它准备到位，所以常常为了让事情往前走，随手给一张干瘪的工单扣上"一周额度"——结果逼着我们边干边现场摸索范围，而不是真正把范围打磨出来。

因为我们是一个接一个地串行写 one-pager，工程师做完手头的项目，往往发现没有打磨好的活在等他。他们只好踩刹车，起草一份 one-pager，等异步评审，等的间隙顺手捡几个小"速赢"。真正要紧的事，势头就这么一次次熄火。

最后，对于周期中途调整优先级，我们也始终没有一套像样的机制。新活进来，我们要么扑上去，要么扔进 Linear，盼着以后能轮得到它。

把 Shape Up 做了一半，我们造出了一把勺叉——喝汤是比叉子强，但我们想要的不止于此。

## 全面押注

我们的解法是真正彻底执行 Shape Up：一个用于猛攻开发的活跃周期，接一个用于维护、规划和准备的冷却期。Basecamp 默认的节奏是六周冲刺、两周冷却。这个比例对他们或许管用，但对一家 8 人的初创公司，六周一锁太长了。我们把它砍半，改成三周活跃周期接一周冷却。

冷却周不是放假，而是用来做准备的。工程师扮演 Shaper（塑形者），就下一个活跃周期的 one-pager 与创始人异步打磨。意愿额度在这一周里被反复讨论、最终锁定。冷却周结束时，我们手上就有一批打磨好的赌注（也就是项目）整装待发。

每个活跃周期的规划会，变成了一场简短的 *Betting Table*（下注会）。我们审视打磨好的 one-pager，决定接下来三周给哪些赌注投钱。一散会，团队接下来一个月就再没有阻塞了。活跃周期内，没人需要起草 one-pager，也没人需要争论范围。我们只管造。

## 零和博弈

到这儿你大概在想，对一家初创公司来说，把计划锁死三周也太长了。你没想错。要让它行得通，唯一的办法是把这个周期当成一场零和博弈。

如果周期中途冒出一件超级重要的事，我们就该去做。但我们得明白，做它就意味着周期里的另一件事得掉出去。这把对话从"我们能做这个吗？"（答案几乎永远是"能，技术上能"）转成了"我们该用这件事替掉那件事吗？"。

这正是认知科学再次登场的地方——就一小会儿。打断对发起打断的人来说是免费的，成本却由团队来背。我们管它叫 **Context Tax（上下文税）**，而这个周期把它摆到了明面上。我们不是要逃掉这笔税——有时新来的事确实更重要。我们只是确保自己是有意去交，并且换来的东西对得起这个价。

## 实操中是怎么跑的？

1.  **新想法：** 我们从各处捞想法——和客户的对话、市场调研，或是我们这支有才华、又谦逊的团队与生俱来的创造力。想法进入 Notion 里的想法池，等着被打磨。
2.  **用 one-pager 界定范围：** 在冷却周里，一名工程师拿起某个想法，和创始人及团队异步协作，敲定意愿额度、严格定义范围、识别风险因素。这从一小时到大约一天不等。范围一旦钉死，就可以进入下一步。
3.  **下注会：** 我们审视所有打磨好的 one-pager，决定想把哪些拉进下一个周期。没进这一轮的，就先搁着，等下一个冷却周再来看。
4.  **活跃周期：** 这是最轻松的部分——团队要做的就是埋头开造三周。没有范围争论，也没有规划。要是额度用光了，我们要么削减范围，要么牺牲周期里靠后的某件事来延长这个项目。
5.  **应对打断：** 周期中途冒出了新东西。我们先停一下，问自己："如果想做这个，得放掉什么？" Context Tax 被大声点出来，然后我们做出一个有意识的决定。要么新东西胜出、别的被往后推，要么它就等下一场下注会。

## 实际感受如何

第一个冷却周听上去像是轻松的一周。没有功能开发，也没有冲刺目标。我们能忙到哪里去？

结果忙得很。那些一直没顾上的 bug 修复、那些一直绕着走的技术债、团队惦记许久的 DevEx 改进。那一周我们照样发了三个版本。这里头有种奇怪的、带点负罪感的快乐。我们没造任何新东西，却在为下一个周期打地基。

这个周期和之前那种自由落体般的状态截然不同。我们有计划，每个人都清楚自己在做什么、从哪儿下手。从第一天起就能扎进去。整件事让人感到平静、从容。算不上完美，但也不再手忙脚乱。

我们更快了，因为我们不再频繁地"开始—停下"。工作更好了，因为人们能把一整个问题在脑子里揣上超过二十分钟。决策更好了，因为它们是在我们被逼着拍板之前就已经做好的。

我们并没有变得更清闲。我们也没找到什么暗藏的多余工时。我们只是不再无意间去交 Context Tax，而真要交的时候（因为有时不得不交），我们是有意去交。

## 还有什么难

一个词：拿捏分寸（calibration）。

零和规则逼着我们对范围诚实，但我们并非每次都拿捏得准。有些在下注会上看着挺紧凑的 one-pager，做起来才发现比看上去宽得多。我们正在变好。

另一件难事是判断紧迫性。这套周期逼出了一条清晰的分界：什么是*真的*紧急，什么只是*感觉*紧急。这是个需要判断的活，我们偶尔会判断错。人生本就如此。

我们还在摸索细节，但大方向看起来是奏效的。那种手忙脚乱的感觉没了，活也推进得很快。这才是全部的意义。

变的不是我们交付了什么，而是这样一种感受：当你正想到一半，知道没人会冷不丁把你拽出来。

我们照样拼命干、不断交付、遇上各种意外。但我们有一份大家都参与制定的共同计划。紧急的事落下来，我们知道怎么接。周期结束时，总有一件做完的事可以拿出来指给人看。

说实话，这感觉真的很爽。
