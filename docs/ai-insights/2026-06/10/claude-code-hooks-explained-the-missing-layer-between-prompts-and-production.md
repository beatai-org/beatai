# Claude Code hooks 详解：提示与生产之间缺失的一层

为什么记忆和规则还不够，以及钩子如何在正确的时间强制执行行为。

这周，在使用 Claude Code 时，我发现自己需要确保几个实际工作流程中的行为保持一致。我原以为 Claude *知道*它应该做什么（规则都设置好了，claude.md 文件中也有清晰的指令，代理也已经配置好了）。然而，有时它能完美地执行这些指令，而有时，我却不得不让 Claude 执行一些特定的后续任务。

就在那时，一位同事向我介绍了钩子。

问题不在于记忆，也不在于提示，而在于我竟然依赖一个概率系统来做出确定性的行为。

这正是钩子设计用来填补的空隙。

在这篇文章中，我将详细解释 Claude Code hooks 的本质、工作原理，以及——更重要的是——如何理解它们。它们并非又一个新功能，而是提示信息和生产系统之间缺失的一层，在这个层面上，行为不再是“尽力而为”，而是开始被强制执行。

如果您正在尝试使基于 LLM 的工作流程在实际项目中可靠运行，那么这篇文章正是为您准备的。

## 本博客将涵盖哪些内容

- **为什么钩子会存在？** 为什么 Claude 知道规则是不够的，以及当行为完全由模型判断时，实践中会出现哪些问题。
- **用一个清晰的思维模型来解释钩子是什么。** 钩子作为生命周期控制点：不是记忆，也不是指导，而是系统可以执行、验证或做出反应的明确时刻。
- **实践中最重要的钩子事件。** 实际驱动大多数真实工作流程的一小部分事件，以及如何在不记住完整生命周期表的情况下思考它们。
- **钩子的实际技术工作原理。** 从事件 → 匹配器 → 结构化输入 → 处理程序。
- **高级钩子（当您希望 Claude 进行二次检查时）。** 一些模式中，钩子超越了自动化，开始强制执行推理，使用提示或基于代理的检查。
- **适用于数据科学和机器学习代码库的真实示例。** 从防护措施和通知到延迟验证和可重复性检查，以及可直接转化为机器学习和数据工作流的具体模式。

我们开始吧！

## 钩子存在的意义是什么？

在之前的文章中，我们看到 Claude 可以[记住一些事情（*记忆*）](https://medium.com/data-science-collective/you-probably-dont-understand-claude-code-memory-3cb26955a8ea)并遵循[结构化的指导（*规则*）](https://medium.com/data-science-collective/how-claude-code-rules-actually-work-8b01faaa4d9d)。这已经足够了。你只需定义它的行为方式，在很多情况下，它就能完全按照你的要求执行。

但是，如果您[看过我的 LLM 系列文章](https://medium.com/data-science-collective/llms-explained-part-1-the-3-layer-framework-behind-chatgpt-friends-6189c3fa8ae3)，您就会知道 LLM 是概率性的，Claude 也不例外。它会根据上下文、模式和权衡取舍来决定下一步的行动。即使它“知道”某个规则，在合适的时机应用该规则仍然是一个决策过程，而不是一种保证。

这种区别比乍看之下更重要。主要是因为两者之间存在*巨大*差异：

- 一个*通常运行正常*的系统
- 以及一个某些事情*总是发生*的系统

Anthropic 自身的文档对此区别做了非常明确的说明：[**hooks 的设计目的是在生命周期的特定节点引入*确定性的***](https://code.claude.com/docs/en/hooks?utm_source=chatgpt.com) [**控制**](https://code.claude.com/docs/en/hooks)，而不是依赖模型来记住或优先考虑正确的行为

这就是钩子的作用所在。钩子的存在是因为有些行为根本不应该依赖于判断。

## 用一个清晰的思维模型来解释钩子的本质。

我们简要介绍了钩子可以在生命周期的特定节点上实现**确定性控制**。关键问题是：*这在实践中究竟意味着什么？*

可以将 Claude 的操作想象成一系列步骤（也就是一个生命周期）：提交提示、调用工具、编辑文件、生成响应、将任务标记为已完成。

这些时刻中的每一个都是一个**事件**。而钩子允许你将逻辑附加到这些事件上。

### 一个简单的例子

假设你想强制执行这条规则：

> *“代码更改后，必须更新 README 文件。”*

你可以把这段文字写在 `CLAUDE.md` 文件中。

但是，问题在于：你依赖模型来**记住并优先执行**这条规则。有时它会做到，有时则不会——尤其是在情境扩展、任务变得复杂或注意力转移时。

理想情况下，我们希望在“代码更改完成”事件完成后触发一个操作。换句话说，我们希望 LLM 能够保证不仅执行更改，还会检查文档并进行相应的更新。

## 实际应用中最重要的钩子事件。

到目前为止，我们一直在讨论“生命周期事件”，也就是钩子引入确定性行为的时刻。但是，当然，最重要的问题是：***我们实际可以使用哪些事件？***

Anthropic [提供了一个完整的生命周期图和事件表](https://code.claude.com/docs/en/hooks?utm_source=chatgpt.com)，内容相当全面。截至撰写本文时，共有 22 个可触发的事件。您绝对*不*需要记住所有事件。我个人觉得将它们分为 5 个类别更容易理解：

每个组代表生命周期中的不同阶段：

- 当会话开始或上下文发生变化时
- 在行动发生之前
- 当某个程序正在运行时
- 在某事发生之后
- 当工作即将被视为“完成”时

一旦你从这个角度思考问题，问题就从：

> “我应该使用哪个事件？”

到：

> “我希望在工作流程的哪个阶段进行干预？”

### 首先需要了解的五个事件

如果您是新手，可以先专注于一小部分事件（表格中用 ⭐ 标记的事件）：

- **SessionStart** — 初始化上下文并进行设置
- **PreToolUse** — 在操作发生之前拦截并控制它们
- **PostToolUse** — 在执行操作后立即做出反应
- **Stop** — 在响应完成之前强制执行检查
- **TaskCompleted** — 强制执行更严格的“完成”定义

这五项涵盖了大多数实际应用场景：防护措施、验证、质量控制和上下文管理。其他功能通常要么是辅助行为（通知、日志记录），要么是更高级的工作流程（多代理协调、MCP 交互）。

### 回到我们的 README 示例

让我们回顾一下之前的规则：*“代码更改后，必须更新 README 文件。”*

对于这种情况，没有唯一的“正确”事件。这取决于你*何时*想要强制执行它。

- 使用 **PostToolUse** → 文件编辑后立即检查
- 使用 **Stop** → 在返回结果之前执行最终验证
- 使用 **TaskCompleted** → 将其作为完成状态定义的一部分强制执行

即使规则本身相同，每一种选择都会导致略微不同的行为。这就是关键所在：

> *钩子不仅仅关乎你强制执行**什么**，更关乎你**何时**强制执行。*

## hooks 的技术工作原理。

从技术层面来说，钩子非常简单。

钩子以一个**事件**开始，可以选择性地使用**匹配器**来缩小范围，从 Claude Code 接收**结构化输入**，然后执行一个或多个**处理程序**。类似这样的序列：

```
event → matcher → structured input → handler → outcome
```

这听起来很抽象，让我们具体化一下。

### 基本的 JSON 结构

最简单的钩子配置如下所示：

```json
{
  "hooks": {
    "PostToolUse": [. 
      {
        "matcher": "Edit|Write",  
        "hooks": [
          {
            "type": "command",    
            "command": "/path/to/check-readme-after-edit.sh"  
          }
        ]
      }
    ]
  }
}
```

在这个例子中，钩子监听 `PostToolUse` 事件，对其进行过滤，使其仅适用于 `Edit` 或 `Write` 工具，然后在满足该条件时运行 shell 命令。

我们已经在上面介绍了生命周期事件，所以让我们更详细地看一下匹配器、结构化输入和处理程序。

### 匹配器

匹配器的作用是避免一直运行钩子（或者换句话说，就是指定*什么决定*钩子应该运行）。

Anthropic 将匹配器作为正则表达式过滤器使用，其过滤的具体字段取决于事件类型。对于工具相关的事件，例如 `PreToolUse`、`PostToolUse` 和 `PostToolUseFailure`，匹配器会针对 `tool_name` 进行过滤。某些事件（例如 `Stop` 和 `TaskCompleted`）完全不支持匹配器，始终会触发。

这意味着：

- `Edit|Write` 匹配任一工具
- `Bash` 仅匹配 Bash
- `mcp__.*` 可能与 MCP 工具匹配
- 无匹配器意味着对于无匹配器事件，“每次都运行”。

这就是为什么匹配器在实践中如此重要的原因。它们使钩子指令精准而不是杂乱无章。

### 结构化输入

所以，**when**（即 `PostToolUse`）和 **what**（即 `Edit|Match`）告诉 Claude 它应该执行某些操作。但在执行操作之前，Claude（或钩子）需要知道刚刚发生了什么。

当 Claude Code 触发钩子时，它不会仅仅说：“嘿，发生点什么了。”它会向你的脚本发送一个**事件的结构化摘要**。你可以把它想象成一张收据：

- 刚才发生了什么？
- 使用了哪种工具？
- 传递了哪些输入
- 涉及哪些文件

这份“收据”是用**JSON**格式编写的。回顾一下我们的钩子示例，当`PostToolUse`事件触发且匹配器（`Edit|Write`）匹配成功时，Claude 会发送类似这样的内容：

```json
{
  "event": "PostToolUse",
  "tool_name": "Edit",
  "tool_input": {
    "file_path": "src/main.py"
  }
}
```

这是**结构化输入**。

这种结构化的输入可以帮助 Claude 指出具体做了哪些更改，这样你的脚本就不必随机扫描文件、依赖模型记忆或猜测发生了什么。它对事件的**精确、结构化的描述**做出了反应。

### 处理程序

最后，我们来看看事件和匹配器匹配成功后实际运行的代码。Anthropic 目前支持 4 种钩子类型：

- **command** → 运行本地脚本（最常见，最容易上手）
- **http** → 将事件发送到外部服务（API、日志记录、Slack 等）
- **prompt** → 运行基于 LLM 的检查或转换
- **agent** → 将任务委托给另一个代理

实际上，你最先看到的钩子大多是 `command` 钩子，因为它们易于理解和调试。一个简单的 shell 脚本或 Python 脚本通常就足以实现第一波有用的自动化。

### 回到我们的 README 示例

我们上面已经展示了一个例子，其中 `type = command` 的目的是运行 shell 脚本。这里是另一个例子，我们可以使用 `agent` 或 `prompt` 来执行某些操作。

```json
{
  "hooks": {
    "Stop": [
      {
        "hooks": [
          {
            "type": "agent",
            "prompt": "Check if README.md reflects recent code changes. If not, update it accordingly."
          }
        ]
      }
    ]
  }
}
```

### 这个 JSON 文件存储在哪里？

Hooks 使用与 Claude Code 设置相同的 JSON 设置格式，Anthropic 将其记录在 `settings.json` 文件中。它们可以在用户级别、项目级别和托管设置中进行配置；插件也可以在支持的情况下使用相同的结构提供类似 Hooks 的配置。

### 等等，我难道不能直接请 Claude 帮我写 hooks 吗？

当然！而且我本人也是这么做的。

但是，了解结构仍然很重要，因为你需要能够评价 Claude 的作品。

你想查看生成的钩子并询问：

- 这是否与正确的事件关联？
- 匹配范围是否过广？
- 这是观察、阻止还是修改？
- 该处理程序是否真正检查了正确的输入？
- 这个程序在我的电脑上运行安全吗？

真正的技巧在于：不是凭记忆输入 JSON 代码，而是充分了解控制界面，以便正确地进行审查。

## 高级钩子（当您希望 Claude 进行二次检查时）。

目前，你在网上看到的大多数例子都是**命令钩子**，即在生命周期的特定点运行的小脚本。

这的确是一个很好的起点。但这也意味着对钩子的功能理解过于狭隘了。

钩子不仅限于 shell 脚本或简单的自动化。它们还可以触发**提示和代理**，这意味着它们可以为您的工作流程引入额外的推理层（我们在上面的 readme 示例中简要地看到了这一点）。

到那时，钩子就不再仅仅是*自动化*，而是开始成为**思考的检查点**。

### 从自动化到验证

这里有一个很有用的区别。

有些检查是机械式的：

- “`README.md` 文件是否已更改？”
- “测试运行了吗？”
- “这个文件是否被编辑过？”

这些都可以通过脚本很好地处理。

但现实世界中的许多检查并非机械式的，而是语义式的：

- “README 文件是否已更新至最新版本？”
- “这个实现方式是否符合原始要求？”
- “在我们认为这项工作完成之前，是否还有什么重要的遗漏？”

这时提示和代理钩子就派上用场了。

它们允许你这样说：

> *“请在此稍作停顿，再三思考后再继续前进。”*

### 使用提示钩子或代理钩子时会发生哪些变化？

除了运行脚本之外，你还可以：

- 运行**定向提示**以验证特定条件
- 触发**代理**审查、改进或完成工作
- 通过 MCP（Slack、Confluence、API 等）调用外部系统

这使得 hooks 更像是**工作流程的控制层**，而不仅仅是一个触发系统。

### 一些具体的模式

与其列举几十个例子，不如通过几个常见模式来观察其发展过程：

你会开始体会到它的强大之处。它不再像是开发者工具，而更像是**工作流程设计**。

您正在做出决定：

- 系统应在此处暂停
- 它应该验证自己的工作
- 在继续前进之前，它应该在哪些方面进行改进？

由于钩子与生命周期事件相关联，因此这些检查每次都会**一致地**发生。

## 适用于数据科学和机器学习代码库的真实示例。

数据科学工作流程可以借鉴许多在软件工程中行之有效的理念（毕竟，我们最终也是要编写代码的）。但我还想向您展示几个示例，希望能激发您对建模工作流程的一些思考。

### 1）编码规范（典型用例）

最简单也最有用的模式之一是在**工作完成时**运行编码规范。使用 `Stop` 或 `TaskCompleted` 钩子，您可以运行：

- 测试
- linter
- 类型检查

正如我之前提到的，这是大家都在谈论的典型用例。不过，以前我写代码的时候，总是忘记执行 `make lint` 或 `make ruff`，结果导致我不得不重新提交到 Git……这些钩子让我的工作轻松多了。

### 2）自动更新实验文档

文档编写是每个人都认同很重要，但几乎总是被忽略的事情之一。钩子机制让你能够将其融入工作流程。

例如，在训练结束后，您可以触发一个钩子，该钩子可以：

- 提取关键参数
- 记录结果（指标、输出）
- 更新 README 文件或实验日志

这可以通过简单的命令钩子来实现，也可以使用能够更清晰地格式化更新的代理来实现。实际上，如果您使用 Databricks（通过 MCP 或 CLI），您可以读取存储的工件，并确保所有关键指标都已记录、标签已应用等等。

关键在于确保文档质量不再是人工任务，而是工作过程中自然而然产生的副产品。

### 3）机器学习训练运行完整性检查

我个人非常喜欢这个。

除了检查某个任务是否运行之外，你还可以使用**提示或代理钩子**来询问：

- 这些指标合理吗？
- 是否存在过拟合的证据？
- 评估结果是否符合最初的目标？

例如，使用 `Stop` 钩子：

```
{
  “hooks”: {
    “Stop”: [
      {
        “hooks”: [
          {
            “type”: “agent”,
            “prompt”: “Review the latest model results. Check whether the evaluation is sound, whether there are signs of overfitting, and whether the conclusions are justified.”
          }
        ]
      }
    ]
  }
}
```

在实践中，你会将此与一些简单的约定结合起来：

- 仅在执行训练脚本后运行此命令
- 或者当生成指标文件时。

最终成果是一个轻量级的“第二意见”工具，可以集成到您的工作流程中。

虽然不完美，但足以在继续操作之前发现明显的问题。

### 4) 长时间运行工作流的通知

许多数据工作流程都是异步的：模型训练、数据抓取、大型转换……然后可能需要很长时间。请耐心等待。

钩子可以大大简化管理过程。例如，使用 `Stop` 或 `Notification` 钩子，您可以：

- 发送 Slack 消息
- 触发 webhook
- 将完成情况记录到外部系统

当与代理或 MCP 集成结合使用时，此功能尤其有用。

与其手动检查“完成了吗？”，不如让系统告诉你。

## 结语

如果要你记住一个关键点，那就是：

> *钩子**不仅仅**是告诉模型该做**什么**。它们还决定**何时必须发生某事**。*

当你开始从生命周期控制而不是指令的角度思考问题时，行为就会停止，这取决于模型是否“记住”了规则，关键检查也会在应该进行的时候准确进行。

钩子基本上为你提供了一种工具，用于在系统中设置**合适的控制点**。

一旦你理解了这一点，hook 就不再像是一个小众功能，而会展现出它们真正的本质：**构建可靠 AI 系统的控制层。**
