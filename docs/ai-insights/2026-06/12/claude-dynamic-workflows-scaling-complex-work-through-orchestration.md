# Claude 动态工作流：通过编排扩展复杂工作

深入探究 Claude 的多智能体编排框架

![](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/01.webp)

大型语言模型（LLM）在单个任务上已经展现出卓越能力。但现实中的许多问题并非单一任务，而是由多个相互关联的步骤组成的复杂工作流，需要规划、并行执行、验证和迭代改进。比如写一份全面的研究报告、审查大型代码库、或构建端到端应用，通常都要协调多个专门任务，而不只是调用一次模型。

为应对这一挑战，Anthropic 于 2026 年 5 月 28 日推出了 **Claude Dynamic Workflows**——一个多智能体编排框架，让 Claude 能把复杂目标拆解为更小的专门任务，协调多个智能体，再把它们的输出综合成一个连贯的最终结果。

动态工作流不再依赖单次模型调用，而是把 Claude 变成一个能像人类团队协作那样、管理复杂流程的系统。

### 本文目标

本文剖析 Claude 动态工作流的架构：它在 Claude Code 中处于什么位置、多个智能体如何协同解决复杂任务，以及设计可扩展多智能体系统的最佳实践。

读完本文，你会明白：

1. **动态工作流在 Claude Code 中的适用场景**，以及何时编排比单智能体方案更合适。
2. **动态工作流的实际运作方式**，从任务分解、并行执行到结果综合。
3. **如何在实践中应用动态工作流**，包括设计模式、最佳实践和常见陷阱。

本文以一个深度研究工作流为例，它要回答一个有争议的研究问题、并写出一份完整报告。这个例子会展示多个智能体如何协作，大规模地调查、分析、验证和综合信息。

> ***先决条件：*** *无需任何前置知识；熟悉 Claude 或 LLM 概念会有帮助。*
>
> ***工具和库：*** [*Claude Code*](https://code.claude.com/docs) *v2.1.154 或更高版本*

运行用的产物和脚本可在 GitHub [这里](https://github.com/linafaik08/claude-code-dynamic-workflows)找到。

## 1. 从单个智能体到工作流：理解协调层级

只有当单个智能体的局限暴露出来时，工作流才有意义。关键区别主要不在模型能力——原则上，一个足够强的智能体可以走完复杂研究任务的每一步。真正的挑战是：如何在很长的时间跨度里保持专注、保证质量，并在不撑爆单个上下文窗口的前提下协调大量工作。

Claude Code 已经提供了多种机制，把智能体的能力扩展到单条提示之外。工作流是其中最具结构性的，但并非首选工具。要理解它的作用，得先理解从单个智能体到越来越显式的协调形式这一演进过程。

### 1.1 单个智能体的局限

单个智能体可以在同一个上下文窗口里搜索、读取、分析、写入。对许多任务，这已经够用。

但随着范围扩大，三种失败模式会反复出现 [2]。

- **智能体惰性。** 智能体在工作真正完成之前就停下。它收集到足够证据、拼出一个看似合理的答案，便宣布成功——哪怕搜索空间的重要部分还没探索 [2]。
- **自我偏好偏差。** 同一个智能体既生成输出、又评估其质量。摘要往往对生成它的系统更有说服力，于是自评不如独立评估可靠 [2]。
- **目标漂移。** 长时间运行的任务会逐渐偏离最初目标。经过多轮总结、上下文压缩和中间推理，原本明确的要求会变得隐晦，甚至彻底消失 [2]。

仅靠提示很难消除这些问题，因为它们源于任务本身的结构。于是问题变成：怎样在不靠堆提示的前提下，克服这些局限？

### 1.2 Claude Code 中协调工作的四种方法

Claude Code 提供了四种逐级增强的方式来协调多步骤工作 [1]。

- **技能（Skills）封装可复用的专业知识。** 它把能在不同任务间反复套用的指令、惯例和最佳实践记录下来 [1]。详见我之前的文章：[构建 Claude 技能：与 LLM 互动的新范式](https://aipractitioner.substack.com/p/building-claude-skills-a-new-paradigm)。
- **子智能体（Subagents）提供委托。** 父智能体可以把一项聚焦的工作交给一个工作智能体，等它做完再收回结果 [1]。
- **智能体团队（Agent teams）引入监督。** 由一个主导智能体协调多个长时间运行的同伴，分派工作并管理它们之间的协作 [1]。
- **工作流（Workflows）把协调搬进代码。** 步骤顺序、分支逻辑、重试和数据流由脚本显式定义，而不是在对话里临场管理 [1]。

关键区别不在于用了多少个智能体，而在于"计划"存放在哪里。

用技能和子智能体时，计划仍留在 Claude 的上下文窗口里：智能体决定下一步做什么，并把每个结果纳入正在进行的推理。智能体团队让一个主导智能体去协调多个同伴，从而扩展了这一模式，但协调依然发生在智能体的决策过程里 [1]。

工作流则把编排搬到软件层：由脚本决定下一步运行什么、保留哪些信息、丢弃哪些、以及结果如何在各阶段之间流转。智能体负责干活，工作流掌控全程。

正是这种分离，使得大规模、可重复的协调成为可能。

### 1.3 升级阶梯

这些机制最好理解为一条递进的阶梯，而不是一组相互竞争的选项。

- **直接问就好。** 当任务能轻松装进一个上下文窗口时，单条提示仍是最简单、最便宜的方案 [1]。
- **技能。** 当同一套标准或流程要反复套用时，可复用的专业知识就有了价值。
- **子智能体。** 当独立的调查会把主线搅乱时，委托就派上用场。
- **智能体团队。** 当多条并行工作需要在较长时间里持续协调时，监督就变得值得 [1]。
- **工作流。** 当协调模式本身就该可复用，或规模大到对话式协调者难以驾驭时，显式编排才有价值 [1]。

每往上走一步，都会增加协调开销、消耗更多 token、也更复杂。所以目标不是追求极致复杂，而是只引入任务所需的那点协调。

![图——Claude Code 中协调工作的五种方法，按其增加的协调程度排序。](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/02.webp)

工作流之所以居于顶端，是因为它解决的问题和其他机制不同：它不是简单地增加智能体数量，而是让协调过程本身变得可编程。

### 1.4 隔离、通信与组合

智能体团队和工作流最重要的区别之一，在于信息如何在系统里流动。

- 在智能体团队里，协调是个主动过程：主导智能体监督多个同伴，工作动态分配，信息可以借由共享的任务结构在各智能体之间流动 [1]。
- 工作流则是另一套理念：各智能体通常彼此隔离，每个只接收一个特定目标、独立完成、返回结构化结果，由工作流脚本负责路由信息、并决定何时把不同分支合并 [2]。

这种隔离不是限制，反而常常是工作流可靠性的来源：独立的研究智能体不会无意间相互带偏结论；审查者可以批判某个成果，而不必背负产生它的那些假设；并行的研究始终保持真正并行，直到专门的综合阶段才合并。

![图示——智能体团队依靠同伴之间的沟通，而工作流则通过脚本协调相互独立的智能体。](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/03.webp)

工作流还能和下层机制自然组合：它可以调用子智能体、套用技能、选择不同模型，并在隔离的工作树（worktree）里运行。反过来，它也能作为技能的一部分打包、分发，让整套编排模式成为可复用的能力 [2]。

## 2. 动态工作流内部：规划、执行与协调

回到前面提出的研究任务：

> *用一个工作流来研究：AI 编码助手是否真的提升了开发者效率。查找近期相关的实证研究，交叉验证它们的结论，并报告共识与分歧。*

乍看只是一个简单的研究请求，实则牵涉多项工作：查找研究、提取结论、比较方法、找出分歧、综合结论。难点不在写摘要，而在如何在保证严谨的前提下协调好这一切。

这正是工作流要解决的那类问题。

### 2.1 工作流触发时会发生什么？

可以在提示里直接提到"*工作流*"一词来显式触发它，就像上面那个生产力研究的例子。

也可以通过 `/effort` 菜单启用 **ultracode** 来隐式触发——这让 Claude 自行判断任务何时大到、复杂到需要工作流编排 [1]。

![图 — /effort 菜单显示了 ultracode 和工作流支持。](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/04.webp)

工作流生成的第一个产物不是报告，而是一个程序。

普通的 Claude Code 会话里，规划和执行都在同一个上下文窗口完成；工作流则不同：Claude 先生成一个 JavaScript 程序，描述任务该怎么执行——需要哪些阶段、哪些可以并行、验证放在哪、结果如何合并。生成出来的工作流可以在执行前先检查一遍。

脚本成为这次运行的唯一真相源：它包含各阶段、分支逻辑、循环、同步点，以及"生成多少个智能体"的决策。中间结果留在工作流的运行时状态里，而不是堆进对话历史；只有后续阶段需要的输出、或最终答案，才会回到 Claude 的上下文窗口 [1]。

针对这个生产力研究任务生成的完整示例可在[此处](https://github.com/linafaik08/claude-code-dynamic-workflows/blob/main/.claude/workflows/ai-coding-productivity-research.js)获取。它创建了多个独立阶段，分别用于发现研究、提取结论、比较方法、验证结论、生成最终报告；每个阶段都能并行跑多个智能体，而不必用一个上下文窗口来保存所有中间结果。

编排与执行分离，正是工作流能扩展的原因。运行时目前最多支持 16 个并发智能体、总共 1,000 个 [1]。

执行开始前，Claude 会把生成的流程拿出来征求批准。审批卡会概括计划的各个阶段、标出可能的 token 消耗，并提供一个"查看原始脚本"选项，用来检查生成的 JavaScript 代码 [1]。

![图 — 工作流审批卡，显示阶段、令牌警告和查看原始脚本。](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/05.webp)

脚本本身不能直接碰文件、shell 或工具，而是协调各个智能体代它去做。工作流是协调者，智能体干实际的活。万一执行中断，恢复同一会话就能让工作流从断点续跑，而不是从头再来。

### 2.2 工作流与静态工作流、`/goal` 和 `/loop` 相比如何？

有三个功能和工作流很像，值得区分一下。

**静态工作流**

- **静态工作流**有预定义的结构。无论用 Agent SDK、`claude -p`、LangGraph、CrewAI 还是自定义编排代码搭建，步骤顺序、分支逻辑和智能体交互都是事先定好的，可在各个任务间复用。
- **动态工作流**则针对当前任务现场生成。Claude 不是从预定义编排里挑一个，而是专为要解决的问题创建编排，生成一份为该请求量身定制的 JavaScript 工作流 [1][2]。
- **举例**：回答这个生产力研究问题时，静态工作流可能永远生成同一套"搜索、分析、综合"的步骤；动态工作流却能临场决定，比如启动多少个搜索智能体、要不要追加验证、综合之前如何对发现分组。

`/goal` 和 `/loop`

`/goal` 和 `/loop` 解决的是完全不同的问题。工作流定义工作**如何组织与执行**；`/goal` 定义工作**何时**算完成；`/loop` 定义工作**何时**该再跑一次 [2]。

- `/goal` 设定一个完成条件，也就是 Claude 必须达到的"完成"定义。
- `/loop` 能按计划、或直到满足某条件为止反复执行任务，因此很适合定期审查、监控或验证流程。

这些功能是对工作流的补充，而非竞争：一个工作流可以协调数十个研究智能体，`/goal` 可以要求达成一致意见才算完成，`/loop` 则能在有新研究出现时定期重跑整套流程。

换句话说，工作流是流程，`/goal` 是完成标准，`/loop` 是重复机制。

### 2.3 驱动多数工作流的六种模式

工作流虽是动态生成的，却通常由少数几种反复出现的协调模式拼成 [2]。这些模式很有用，因为它们给了你一套描述"工作如何组织"的词汇——一旦任务的结构清晰，往往可以直接在提示里点名要哪种。

**1. 分类并处理。** 由轻量级智能体给每个条目分类，再路由到对应的处理程序。

![](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/06.webp)

*例如：一个收件箱工作流，把每张工单标记为缺陷报告、退款请求、升级咨询或垃圾信息，再转给专门的智能体处理 [2]。*

**2. 扇出并综合。** 把一项任务拆成若干独立部分，每个智能体在各自的上下文里处理一部分，再由一个综合步骤把结果合到一起。

![](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/07.webp)

*例如：生产力研究工作流——多个智能体并行检索实证研究，再由综合阶段把发现汇成一张证据图谱。尽职调查工作流也是同一模式：不同智能体在最终报告汇总前，各自审查数据室的不同部分 [2]。*

**3. 对抗性验证。** 一组智能体产出结果，另一组则按规则设法挑它们的刺。

![](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/08.webp)

*例如：生产力研究工作流的验证阶段——审查智能体核对结论是否有所引研究支撑，并标出薄弱或自相矛盾的说法。给每条主张派一个验证者，同样可以用来核查稿件事实 [2]。*

**4. 生成并筛选。** 先生成大量候选，再评估、收窄。

![](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/09.webp)

*例如：生成几十个标题、设计或方案，再从中挑出最好的几个。关键原则是：生成者和评估者必须是独立的智能体 [2]。*

**5. 锦标赛。** 候选项两两捉对比较，而不是各自独立打分。

![](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/10.webp)

*例如：给数千份简历、提案或生成结果排名。每次比较都独立判断，从而减少长时间评估中常见的偏见与上下文累积 [2]。*

**6. 循环直到完成。** 反复执行，直到触及一个有意义的停止条件。

![](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/11.webp)

*例如：排查一个不稳定的测试——生成假设、在隔离的工作树里验证，一直测到连续几轮都查不出新问题为止。与固定迭代次数不同，工作流会在任务真正穷尽时才停 [2]。*

多数真实工作流会组合好几种模式，而不是只用一种。比如生产力研究工作流：用扇出与综合来收集证据，用对抗性验证来质疑结论，用筛选来判定哪些研究相关。

### 2.4 如何运行、监控和共享工作流？

一些操作细节和编排模式同样重要。

- **工作流在后台运行。** 一旦获批，它便独立执行，主会话照常可用。`/workflows` 视图实时展示进度：当前阶段、各个智能体、token 消耗和工具使用 [1]。
- **执行可暂停、停止、恢复。** 实时视图支持在任意时刻暂停或终止；若执行中断（比如关掉终端），恢复同一会话会从原状态续跑，而非重来 [1][2]。
- **工作流可保存、可共享。** 完成的工作流能导出为 JavaScript 文件；和一个 `SKILL.md` 及任何配套评分标准打包在一起，就能像技能那样分发、复用 [2]。
- **token 用量可控。** 由于工作流并行协调多个智能体，可能吃掉大量 token。可以在提示里指定预算上限，也可以随时从 `/workflows` 视图停掉 [1][2]。
- **并非所有任务都需要工作流。** 它是为大规模、并行、迭代或对抗性的问题设计的；小而线性的任务，往往一条指令更省事。工作流的价值在于它带来的协调，而不在于动用了多少智能体 [2]。

工作流位居顶端，不是因为它在任何场景都最强，而是因为它带来的协调开销最大。

## 3. 实战动态工作流：一个端到端的研究工作流

让我们走一遍*生产力研究工作流*的完整运行过程。

### 3.1 工作流是怎么组织工作的？

生产力研究工作流落地为一张四阶段执行图：**搜索**、**获取**、**验证**、**综合**。这几个阶段合起来实现了"扇出与综合"模式，并在写最终报告之前插入了一道对抗性验证 [2]。

![图示——流程概览。顶部是四个阶段，下方是 66∶1 的来源漏斗，共 27 个智能体，总成本约 5 美元。](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/13.webp)

![图——搜索阶段的 /workflows 视图。七个研究智能体并行运行，各自追踪不同的证据线索。界面显示每个智能体的 token 消耗、工具使用、运行时间和整体工作流进度。](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/14.gif)

实时视图里能看到这套设计的几个侧面。

- **工作流把问题拆成多个阶段。** 左侧面板就是执行计划：搜索、获取、验证、综合。每个阶段必须完成，工作流才进入下一个。
- **搜索阶段扩展到七个并行智能体。** 工作流没有让一个智能体去泛泛地调查 AI 编码助手，而是把不同的研究角度分给不同的智能体。
- **每个智能体独立运行。** 中间面板显示正在跑的研究智能体，右侧面板跟踪 token 消耗、工具使用和执行时间。由于各自拥有独立的上下文窗口，它们能探索不同证据而互不干扰。
- **工作流负责协调，智能体负责干活。** 编排层决定启动多少智能体、跟踪它们的完成情况，并等所有搜索任务都结束才进入下一阶段——这正是"扇出与综合"模式的同步屏障特征 [2]。

搜索阶段结束后，工作流进入更深入的分析。

- **Fetch 把来源集收窄到最相关的研究，并提取结构化证据。** 对每篇选中的论文，都分析了方法、样本量、报告的效应、局限和资金来源。
- **Verify 引入一位独立审核者。** 一个专门的验证智能体交叉核对各研究的结论、查找矛盾，并重点指出哪些结论取决于方法或研究设计。
- **Synthesize 生成最终报告。** 综合智能体拿到的不是"摘要的摘要"，而是前几个阶段收集到的丰富证据，据此产出最终分析。

![图——经过搜索、提取、验证和综合等工作流程后生成的完整研究报告。](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/claude-dynamic-workflows-scaling-complex-work-through-orchestration/15.webp)

### 3.2 验证阶段增加了哪些价值？

这正是工作流值回那些额外 token 的地方。单个智能体审查自己的工作，容易陷入自我偏好偏差；独立的验证智能体则不会 [2]。

**处理相互矛盾的结果。** 验证阶段揭示出文献里一个真实的冲突。Becker 等人（2025）针对在成熟代码库里工作的资深开发者做了一项独立研究，发现 AI 辅助导致**19% 的速度下降** [6]；Peng 等人（2023）则在一项更简单的编码任务上、围绕 GitHub 做研究，结果显示**速度提升 55.8%** [5]。工作流没有把两个结果一平了之，而是把这种差异当成需要解释的问题。最终报告指出，差异来自任务复杂度、开发者经验和研究设计的不同。

**发现系统性规律。** 交叉核对还揭示出一种"资金规律"：最亮眼的正面结果往往来自与厂商相关的研究，而独立研究更常报告对资深开发者是混合、无效甚至负面的影响。报告点出了这一规律，但没有因此否定任何一方的证据。

**保留反直觉的发现。** 最后，验证阶段守住了最令人意外的发现之一：在 Becker 等人的研究里，开发者自认效率大幅提升，实际测得的生产力却在下降 [6]。主观感受与客观表现之间的落差，成了一个关键结论，而没有在综合阶段被当作细节抹掉。

这些洞见之所以浮现，是因为有一个智能体的任务就是质疑别人的结论，而不是顺着把它们延伸下去。

### 3.3 这种方法的成本和局限性是什么？

这个工作流生成了 27 个智能体，在 463 次工具调用中消耗约 807,000 个子智能体 token，耗时约 25 分钟。用 Sonnet 4.6，仅子智能体的成本就约 5 美元。审批卡上的 token 警告不是摆设：工作流的成本可能远高于一次普通对话 [1]。

有三点需要注意：

- **这些来源是检索来的，不是精选的。** 工作流只能分析它找得到的研究。
- **结构化输出能提升可靠性，却提升不了正确性。** JSON 模式能减少提取环节的偏差，但不保证论文被正确解读。
- **结果只是当下的一张快照。** 下个月再跑一遍，可能找到不同的研究、得出不同的结论。

更大的启示，与其说关于 AI 编码助手，不如说关于工作流设计本身。工作流在这里之所以有价值，是因为问题本身有争议，需要对证据加以比较、质疑和综合；要只是查个事实，同一套机制只会徒增成本、带不来新洞见。这把话题拉回第一章那条协调阶梯：当难点在于协调工作、而不只是生成答案时，工作流才最有用。

## 要点总结

✓ **单个智能体的局限通常是结构性的，而非提示层面的。** 随着任务规模扩大，智能体惰性、自我偏好和目标漂移会自然冒出来。工作流把工作分给拥有独立上下文、职责明确的专门智能体，从而缓解这些问题。

✓ **工作流应被当作最高一级的协调机制，而非默认选项。** 技能、子智能体、智能体团队和工作流，构成一条复杂度与开销递增的阶梯。当任务需要跨多个步骤做大量并行处理、验证或协调时，工作流的价值才显现。

✓ **动态工作流把编排从对话搬进代码。** Claude 不再靠单一上下文窗口去协调，而是生成一份 JavaScript 工作流，其中定义了各阶段、分支逻辑、同步点和智能体分配。这种分离让复杂流程能扩展到单个对话难以驾驭的规模。

✓ **少数几种协调模式撑起了多数工作流。** 分类并处理、扇出与综合、对抗性验证、生成与筛选、锦标赛、循环直到完成——这些模式提供了可复用的积木，能组合起来解决更复杂的问题。

✓ **工作流的主要优势，在于协调与验证。** 在生产力研究这个例子里，价值体现在独立审查证据、解释相互矛盾的发现、识别系统性规律、保留反直觉的结果。换成更简单的任务，同一套编排只会徒增成本和复杂度，带不来实质收益。
