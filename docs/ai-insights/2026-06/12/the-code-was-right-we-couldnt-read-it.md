# 代码没错，但我们看不懂。

如今在工程团队——包括我自己的团队——里，常听到这样一句话。

我们其实没仔细检查过，但它确实有效。

一年前，这话会让我停下来；现在却能轻易略过。大约过去六个月，情况变了：软件领域最古老的安全网——人总能读懂代码——失灵了。不是因为代码变得难读，而是因为"跳过代码"变得可以接受。没人规定该如此，它只是悄无声息地不再是铁律——一个个看似合理的决定，慢慢就把规则改了。

![软件领域最古老的安全网失效了。但这并非出于任何人的决定。](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/the-code-was-right-we-couldnt-read-it/01.webp)

我想认真对待这件事，因为这不是纪律问题，而是一种新的抽象——我们接受它时，却没拿对待以往一切抽象的标准去检验它。

先说清一点，因为这才是关键：代码并非不可读，任何一段都能打开看懂。失去的是把握整个系统的能力——看不清它整体如何运转。每个局部依旧清晰，拼到一起的整体却读不懂了。

## 我们过去能够检查的内容

工程一直在做一种交换：把不再需要手工做的事让出去。我们不再写汇编、不再管理内存、不再读编译器生成的机器码。每一次都让渡了一些可见性，但回报丰厚。

而每一次交换都建立在同一个底层前提上：停掉的检查，本就停得安全。计算器每次返回相同结果；编译器是确定性的，需要时还能查它的行为、追它的逻辑。我们停止检查，不是因为查不了，而是因为没必要查——抽象层底下的活儿已经定型，正确验过一次，此后永远照同样的方式运行。

正是这种"安静"让一切成立：我们移除的，是不必再检查的工作。

而人工智能拿走的，恰恰是我们仍然需要能够核查的那部分工作。

倒不是全部。agent 产出的很多东西尽可放心撒手——测试、迁移、胶水代码等等，就像当年撒手汇编一样。麻烦的是你仍需理解的那部分，而它不会主动表明自己和其余有什么不同。

它也不会发出明显的报错信号。编译器出错，通常留得下显眼的痕迹；而人工智能写错的代码，往往和正确代码长得一模一样——能编译、能过那些一眼能想到的测试，读起来就像一个称职工程师写的。错的是判断，不是语法，而判断不会显示在 diff 里。

量还大到任何评审流程都扛不住。于是我们停了评审，理由很简单：太多了，而且大部分确实没问题。可这么一来，得到的是一种全新的抽象——偏偏丢光了旧抽象里那些让我们能安心停手的特性。

## 看起来很安全的模式

麻烦正是从这里来的。

当时我们在搭一个很常规的系统：核心系统触发事件，把事件连同各种数据属性，同步给一个处理客户消息的第三方平台。这套流程是可复制的——先接好一个事件、验证它有效，再对其余所有事件如法炮制。

这样的事件有几十个，每个不过是同一个想法的细微变体。于是我们用了聪明办法：先完整搭好一个、确认模式成立，再让一个 agent 把其余的补完。既然每个都和上一个高度相似、而第一个又是对的，我们就没再细审其余那些。

每一处都完全正确：agent 忠实复刻了模式，每个事件都同步了，每个属性都落在该落的位置。随便挑一处去查，你都会签字放行。

但这个 agent 从不做整合。一个人做到第十个、第二十个、第四十个相同的东西时，会开始吃不消，转而想找一个中心把信息收拢起来。这种本能无关整洁，而是明知以后还要回头理解时的自保。可 agent 没有"以后"：它生成第四十个实例和生成第一个一样快，随手就地接线。结果，事件同步散落在整个系统里，你根本找不到一处能把它看明白。

一切照常，什么都没坏，功能全都正常。可我们却没法靠检查回答一个最基本的问题：这里到底要发出什么、何时发出？

## 易读性是整体的属性

![各部分都正确。但整体而言难以理解。](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/the-code-was-right-we-couldnt-read-it/02.webp)

这正是我反复强调的区别：正确性是局部的，可读性是全局的。

哪怕每一处改动都验证过，也可能没察觉整个系统已经变得无法解释。任何正确决策都担保不了整体的清晰，因为整体从来不是某个单一任务能解决的。没人负责整合，于是没人去做，而每一个正确的局部，都让整幅图景更难看懂。

接着，可读性不再是抽象概念。几周后回来加新事件，发现有些事件的触发时机完全不合逻辑。调试就得手动重建：哪个事件在哪儿发、发了什么——这恰恰被先前散落的结构卡住了。过程中又冒出第二个问题：agent 对发送时机做了错误假设，误读了业务上下文，而这种误读，单看一处 diff 根本发现不了。

真正的问题是路由错了，可正是"读不出来"这一点把它藏住了。要是所有数据都走同一条管道，那个发错的事件早就明晃晃摆在那儿。可读性不是系统跑通之后再加的锦上添花——它恰恰是发现下一个问题的前提。

## 人为什么会去做整合？

![它从不生气，也从不回来。](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/the-code-was-right-we-couldnt-read-it/03.webp)

人们很容易把这种整合本能叫做"优秀工程"。我却不这么看——它其实是一种着眼长远的自利。

当你明知一样东西将来归你所有，构建方式就会不同：你会在深夜调它，会在半年后扩展它，会在别人追问它为何这样运行时负责解释。正是这种"未来感"塑造了你的活儿——你让系统易懂，因为你自己非懂不可。可读性不是你额外赏给代码的优点，而是"预期要长期持有"自然带出的结果。

这也是外包开发很难做到位的原因之一：写代码的人不是最终要扛它的人，缺了那股默默维护可读性的动力，而这股动力没有任何流程能完全替代。

人工智能把这一点体现得最为纯粹：它对"明年要维护的代码库"毫无利害关系，因为对它而言明年并不存在——每次调用都是全新的开始。它不会因为要理顺自己写的代码而烦躁，每次都毫不费力地配合。你一指出代码乱，这些工具的反应就暴露了这种缺失：立刻认同、动手修、也不会把毛病带进下一个文件。没有什么逼它们写得更易懂，因为没有什么让它们为代码的缺陷付出代价。

## 手动把激励放回原位

那么，对于过去白送、如今不再白送的那份"财产"，该怎么办？

不必逐行去读。那条线已经消失，也不会回来，假装它还在只是怀旧。大多数生成的代码可以不读，就像没人会去读汇编。关键从来不是读了多少，而是让哪些东西保持清晰：数据怎么流动、业务意图编码在哪、哪些决策一旦重建代价高昂。如今的活儿，是刻意守住这些，因为写代码的那个程序不会替你守。

第一步，建立所有权：指定专人负责解释系统，而不只是交付。所有者能设计出清晰的系统，是因为预料到会被追问"它为什么这样运行"——把这种预期重新装回去，就能找回大半的直觉。所以构建时，主干结构由人来定：你搭好中心管道，agent 把各个组件接上去。这不是因为"设计先行"多高尚，而是因为那个生成组件的东西，根本没有理由把自己收束成单一形状。

第二，把"可解释"当成系统的真实属性，而非锦上添花。一段代码若没人能讲清它为何这样运行，哪怕本身正确，也是缺陷。发现哪里读不懂，就像修破窗一样随手补上，不必大动干戈。对那些答错代价高昂的部分，老办法依然管用：把人叫到一起，让他对着代码大声讲这段是干什么的——讲不出来，问题就在那里。

第三，别只为 agent 优化。如今流行一种说法：目标是建一个对 agent 友好的代码库。这话只对了一半。代码现在有两个读者：agent，以及要为代码负责的人。我们要的不是一个人工智能用着顺手的代码库，而是一个在人工智能处理过每一部分之后、人依然看得懂的代码库。只为第一个读者优化，恰恰就犯了本文要讲的错。

## 它的代价，以及何时显现

![第四十份拷贝曾经让我感到痛苦。那种痛苦是一种警告。](https://cdn.jsdelivr.net/gh/genesislab-io/beatai-assets@a5fed6379ff4651182f71e4aa23461de93283e89/ai-insights/2026-06/12/images/the-code-was-right-we-couldnt-read-it/04.webp)

我们这次及时发现了。代价是一个一头雾水的下午加一次重构，谈不上事故。但那其实是运气，只是装成了"有惊无险"。

我想反复强调：人工智能不是问题的根源，它只是抹掉了过去那道提醒我们留意的摩擦。手动写第四十遍几乎一样的东西很烦人，而这股烦躁，恰恰是该停下来做整合的信号。我们过去爱把结果归功于判断力，可很多时候，不过是有人不想再忍第四十一次的烦躁罢了。

过去有两样东西几乎不花成本就守住了可读性：一是近处手动重复带来的摩擦，二是远处总有人要为结果负责。这两样，agent 全都感觉不到——它既不会烦，也不必重复劳动。

于是，清晰不再自动维持。它需要你刻意去守，否则只能眼看着它被一个个看似合理的决定一点点蚕食。
