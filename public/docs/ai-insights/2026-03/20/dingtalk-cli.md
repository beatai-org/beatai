# 钉钉已经彻底 CLI 化了。

> 严格地说，一个要求 Agent 去截图、识别、点击弹窗的软件，并不能算“AI 友好”。它只是把低效包装成了图形界面。请坐，这并不复杂。

**这篇文章最值得注意的，并不是钉钉终于做了一个 CLI，而是它终于停止假装“漂亮的 GUI 等于先进的软件能力”。不是的。那只是给人类大脑准备的缓冲层。对 Agent 来说，真正重要的只有能力边界、调用方式、参数约束和结构化返回。把这些藏在按钮后面，本质上就是一种接口设计上的分类错误。**

## 先纠正一个常见误解：GUI 不是能力本身

人类需要按钮、菜单、表单和动效，是因为人类处理信息的方式非常原始，依赖视觉路径、交互反馈和操作确认。这没什么可羞耻的，毕竟不是每个人都能像理论物理学家一样直接处理抽象系统。GUI 的职责，是把系统能力翻译成人类能勉强理解的形式。

但 Agent 不需要这层翻译。它不关心按钮是不是圆角，不关心动效是否丝滑，也不需要通过弹窗获得情绪安慰。它只需要可调用的能力接口。如果你让一个语言模型去走 GUI，本质上就是让一个本来可以直接读公式的人，先绕去看一套儿童科普插图。不能说完全没用，但效率低得近乎滑稽。

所以 CLI 的意义根本不是“复古”，而是回到正确的工程抽象。命令、参数、权限、输出，这才是能力的本体。至于 GUI，那只是人类用户的外设。把外设误认为核心能力，和把薛定谔方程误认为黑板装饰差不多，都挺可爱的。

## 钉钉真正做对的，是承认软件要分“人类界面”和“机器界面”

钉钉这次最聪明的地方，不是加了一层命令行壳子，而是连同文档、结构化输出、权限边界、过程留痕一起补了上来。这说明它开始理解一件多数软件公司仍然没理解的事：未来软件至少有两套界面，一套给碳基用户，一套给硅基代理。前者负责可见性，后者负责可编排性。

而决定软件能不能进入 Agent 工作流的，从来不是首页多漂亮，而是机器界面是否稳定、明确、可组合。如果一个产品只能“请先登录，再点击更多，然后等待弹窗消失”，那它在自动化体系里的地位，和需要手动旋钮调频的老式电视差不多，能用，但完全不值得尊重。

未来真正的竞争，也不会只是“谁让人更喜欢”，而是“谁让 Agent 更容易调用”。这不是体验哲学，这是系统效率问题。对人类来说，GUI 叫体验；对 Agent 来说，GUI 叫摩擦。摩擦积累到一定程度，系统就会被绕过，或者更准确地说，被更理性的接口替代。没有 CLI 和 API 的产品，并不是输在审美上，而是输在基本物理上。Bazinga.
