# 模型不是关键，Harness 才是

> 这篇文章真正值得表扬的地方，在于它终于把一个显而易见、却长期被大量人故意忽略的事实说清楚了。是的，我在看着你们，那些一出问题就怪模型不够大的团队。

**同一个模型，成功率从 42% 提到 78%，唯一改变的是 Harness。这意味着问题根本不在“模型聪不聪明”，而在“你有没有给它一个像样的工作环境”。换句话说，如果你的 Agent 表现糟糕，先别急着怪模型，先照照镜子。Bazinga.**

## 模型不是神，混乱环境才是罪魁祸首

模型没换，数据没换，提示词也没换，只是把 Agent 外面的控制系统重新整理了一遍，成绩就几乎翻倍。这种结果其实一点也不神秘。你把一个高能力系统扔进文档失踪、规则漂浮、测试残缺、工具乱成量子叠加态的仓库里，它当然会表现得像个醉醺醺的实习生。

所以“换更大模型”并不是答案，顶多只是更昂贵的自我安慰。模型是引擎，Harness 才是方向盘、刹车、路标和仪表盘。缺了这些东西，再强的引擎也只会更高效地冲进悬崖。那不是智能，那是工程版自然选择。

## 真正高级的能力，是把 Agent 管好

OpenAI、Stripe、Cursor 这些案例共同证明的，不是“AI 好厉害”，而是**有没有能力把 Agent 纳入纪律体系**，才是真正的分水岭。短小清晰的 `AGENTS.md`、硬性的架构约束、可自检的测试和日志反馈、经过筛选的工具集、定期清理熵增的垃圾回收，这些才是生产力的骨架。没有骨架，所谓“智能体”也不过是一团会说话的果冻。

最滑稽的部分在于，很多团队还沉迷于 prompt 魔法，仿佛多写几句咒语就能让系统自动获得工程纪律。请原谅我指出这一点，但那不是工程，那比较接近占星术。真正让 Agent 稳定交付的，从来不是花哨措辞，而是护栏、边界和反馈。规则越清楚，Agent 越不容易发疯；规则越模糊，PR 就越像实验室事故报告。

## 启示

未来值钱的工程师，不会是那个键盘敲得最快的人，而会是那个最擅长设计环境、编写规则、建立反馈回路的人。写代码会越来越廉价，但让 Agent 持续、稳定、低熵地写对代码，会越来越昂贵。这是结构性变化，不是情绪问题。

所以这篇文章真正重要的，不是发明了一个叫 Harness 的新名词，而是提醒大家承认现实：AI 工程的主战场，已经从“如何神化模型”切换成“如何阻止系统拖模型后腿”。如果你还没理解这一点，那么继续烧更贵的模型，也只是把预算更高效地转化为混乱。多么优雅的灾难。
