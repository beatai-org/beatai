# AI PM 的菜单：成本-质量权衡实战指南

![一份帮助 AI PM 更审慎地打造 AI 原生产品的菜单](./images/the-ai-pms-menu-a-field-guide-to-cost-quality-tradeoffs/01.webp)
*一份帮助 AI PM 更审慎地打造 AI 原生产品的菜单 :)*

我经营一家创业公司。打造 NomNow 这个已经上线 App Store 的多模态 AI 营养师，让我学到了很多东西。但如果只能让年轻时的自己读到其中一条，我最想说的是：**对一家基于 LLM 的创业公司来说，成本纪律不是优化，而是生存。**

在传统 SaaS 里，每个用户的边际成本接近于零。你可以粗放很久，直到它真的成为问题。但在 LLM 里，边际成本就是你设计架构时做出的选择，而且这笔成本并不小。一个消费级 AI app 如果把每条消息都路由到最强的前沿模型，每个活跃用户每天可能花掉 0.50 到 2.00 美元。只要规模稍微有意义，这就是整个生意本身。

所以，当我开始设计 NomNow 时，在写下第一行 agent 代码之前，我先做了一个决定：**路由图应该长什么样？**

诱惑是把所有东西都发给 GPT-4o。多模态记录在它上面效果非常好：午夜拍的一张炸酱面照片、一段含糊不清的语音备忘、"我吃了昨天吃的那个"。但如果连简单的东西也发给它，比如"我达标了！"这种消息、"豆腐里有多少蛋白质"这类问题、跑题闲聊，就等于为规则引擎都能解决的问题支付前沿模型的价格。

![](./images/the-ai-pms-menu-a-field-guide-to-cost-quality-tradeoffs/02.webp)
*左：鼓励类消息交给规则引擎。右：多模态记录交给 GPT-4o（当时的 SOTA）。*

所以我做了路由。多模态记录交给 GPT-4o（当时的 SOTA）。纯文本营养问答交给一个带检索能力的更便宜模型。鼓励类消息交给规则引擎，零 token。跑题内容给出温和引导，同样零 token。用户体验不变。平均成本大约降低一个数量级。

![](./images/the-ai-pms-menu-a-field-guide-to-cost-quality-tradeoffs/03.webp)

这个决定，以及后来几十个类似决定，就是这篇文章想讨论的主题。**对 AI 产品来说，成本不是 PMF 之后再修的东西。它是产品的一部分，必须从第一天起就被设计进去。**

## 为什么这是产品决策，而不是工程决策

有两个原因。

第一，不同实现之间的成本差距非常大。同一个功能，用两种方式做，token 成本可能相差 30 倍或 50 倍。这不是"优化"，这是"可行还是不可行"。事后再怎么补救，也弥合不了这个差距。

第二，每个成本旋钮也都是产品旋钮。用自我批判来重试？会增加用户能感知到的延迟。路由到更小的模型？会改变语气和准确性。用固定 workflow 代替开放式 agent？会改变产品*能做什么*。这些不是后端决策。它们是披着工程外衣的产品决策。

因此，拥有产品 UX 的人，也必须拥有成本形态。默认"所有地方都用最好的模型"是一种决策。默认"工程会解决它"也是一种决策。以我的经验看，这两者通常都是错的。

下面这张图是这个领域里一张很熟悉的图，最初由 Artificial Analysis 汇编，并在 [Zilliz 关于 RouteLLM 的文章](https://medium.com/@zilliz_learn/routellm-an-open-source-framework-for-navigating-cost-quality-trade-offs-in-llm-deployment-7c4ee2158835)中复现。它用一张图概括了局面：不存在"最好的"模型。存在的是一条前沿曲线，而你的工作是弄清楚产品的每个部分应该落在曲线上的哪个点。

## 单次调用

默认选择。一个 prompt，一个答案。如果你能用一句话描述任务，而答案能放进一段话里，就不要给它加戏。很多产品开始变差，正是从那种"把它做成 agent"的冲动开始的。

## Chain-of-Thought

让模型在回答之前一步一步思考。多花 20% 的 token，在推理任务上换来 10-30% 的准确率提升。**这是菜单上唯一的免费午餐。**如果你的任务涉及任何链式逻辑，就应该打开它。注意：在小模型（<70B）上，它可能让结果变差，因为这些模型会生成听起来很流畅、但没有真正跟住底层逻辑的推理。信任之前先测试。

## 路由

先对请求分类，再把它送到能处理它的最便宜路径。这是我在 NomNow 最倚重的模式。这个直觉也有学术依据：UC Berkeley 在 2024 年的 RouteLLM 论文显示，即便是一个简单的二元路由器，也能在成本降低 2.5 倍时保留强弱模型之间 80% 的质量差距，或者在成本降低 3.6 倍时保留 50% 的差距。如果你把每个查询都发给最好的模型，就是在为平均查询并不需要的能力多付钱。

路由有三个值得跟踪的指标，都来自那篇论文：**命中最昂贵路径的请求百分比**（应该很小，而且你应该清楚知道为什么）、**恢复的性能差距**（你的路由器捕捉到了多少质量差异；完美是 1.0，随机约为 0.5），以及**调用-性能阈值**（给定一个质量目标时，昂贵调用的最低百分比是多少）。这些指标会告诉你，路由到底在做实事，还是只是在增加复杂度。

**路由的主要限制是：它的效果只和你的桶一样好。**用户不会按你的桶说话。他们会写"我吃了两个苹果，那里面有多少糖？"（记录 + 查询同时出现）。他们会写"今天很糟"（闲聊？记录？情绪？）。他们会提出菜谱请求、天气问题，偶尔还有 prompt injection。生产级路由系统需要四类 fallback：一个**置信度阈值**，低于它时系统会请求澄清，而不是猜测；面向跨桶查询的**多意图处理**（本质上是一个小 workflow）；一条礼貌说明自己不能帮什么的**范围外路径**；以及针对 prompt injection 和敏感话题的**安全覆盖**。没有这些的路由只是 demo，不是产品。更深一层的观点是：**当产品范围清晰时，路由才是正确模式。**对 NomNow 来说，"帮助用户记录并理解自己的食物"是一个有边界的问题。对开放式助手来说，路由会崩掉：桶会倍增，分类器会退化，你可能应该改用 agent。

## Workflow

你定义步骤；模型在每一步里完成具体工作。整个流程都能画在白板上。它比 agent 更便宜、更可靠，也更容易调试。**Anthropic 公开发布的建议，也是我认同的建议：如果能用 workflow 解决，就应该用 workflow。**很多本应做成 workflow 的产品，却因为 agent 听起来更酷而被做成了 agent。结果 agent 更差。

## Agent loop

模型在每一步决定下一步做什么。你无法提前把流程画在白板上。当任务真的开放时，它很强大；当任务并非如此时，它昂贵且不可预测。成本方差是杀手：同一个查询，周二的成本可能比周一高 5 倍，只因为模型决定多试几件事。对创业公司来说，这种方差是产品问题，不只是财务问题。

## 解耦规划（ReWOO 及类似方法）

agent 的一种变体：先提前规划完整的工具调用序列，执行所有工具（通常并行），再综合结果。*当计划可预测时*，它比标准 agent loop 便宜约 5 倍。它很适合结构化研究（"比较这三家公司过去四个季度的表现"），但当每一步结果都会决定下一步时就会失效。

*图：ReWOO 的工作方式。来源*

## 自我批判与重试

模型生成 -> 另一次调用批判 -> 再一次修订。重复。成本是 6-10 倍，准确率增加 5-25%，**但前提是你有关于"正确"的客观信号**（测试通过、规则引擎验证、引用核对无误）。没有这种信号时，批判者只是在猜，迭代可能比第一稿漂得更差。还有：6-10 倍的延迟。不要把它放进同步聊天 UX。

## Tree search（Tree of Thoughts、LATS）

每一步生成多个候选，给它们评分，再搜索一棵树。在狭窄、可验证的任务上会有显著收益（数学、带测试的代码）。成本是 10-30 倍。我还没有在消费级产品里上线过这个，而且如果有人说他上线过，我会保持怀疑。它适合异步、高价值、强可验证的任务。

![](./images/the-ai-pms-menu-a-field-guide-to-cost-quality-tradeoffs/07.webp)
*图：Tree of Thoughts（ToT），来源*

## Multi-agent

多个专门模型由一个 orchestrator 协调。Anthropic 公开的数据是成本 15 倍，研究任务质量 +90%。正确用例很窄：**真正并行、彼此独立的子任务。**Cognition（Devin）在 2025 年发布了一篇叫 "Don’t Build Multi-Agents" 的文章，针对编码提出了相反观点。他们关于编码是对的，Anthropic 关于研究也是对的。在动用这个模式之前要问的问题是：*这些子任务真的能在不共享上下文的情况下并行完成吗？*如果可以，也许值得。如果它们需要不断互相确认，你得到的会是一个比设计良好的单 agent 更昂贵、更不可靠的东西。

## 我每周真正会思考什么

知道菜单是必要的，但还不够。更难的是培养从菜单中选择的判断力。为了做到这一点，我发现自己会反复回到一些原则上。这些原则在 NomNow 的各种决策中经受住了考验。

**第一，workflow 几乎总是正确的起点。**不是因为 agent 没用，agent 有用，NomNow 也有 agentic 组件，而是因为错误地认为自己需要 agent 的成本，远高于错误地认为自己需要 workflow 的成本。一个后来发现需要 agent 灵活性的 workflow 可以被扩展。一个后来发现只是过度设计的 agent，要退回来就难得多，因为当你意识到这一点时，你已经上线了一个单位经济模型跑不通、失败模式也没有完全理解的东西。Anthropic 的粗略指引很适合做直觉检查：每个任务 0.10 美元的预算可以买到 30-50K token，对 workflow 来说很充足，对 agent 来说则很紧。如果你的产品一年要做一百万个任务，而你在 workflow 本来就足够时把它设计成 agent，那你可能无谓地烧掉七位数的钱。

**第二，通常有一些你还没用上的免费质量提升。**Chain-of-Thought 是显而易见的一个：在 token 成本上增加 20%，换取推理任务上的两位数准确率提升，这笔交易你每次都应该接受。但还有其他方法。工具描述就是一个被低估的例子：Anthropic 团队发布过数据显示，只重写工具描述，在同一个模型、同一份代码、同样成本下，就能把任务成功率提高 5-20%。prompt 里的 few-shot 示例也是一个。一般规律是，prompt 层的质量提升几乎总是比架构层的质量提升便宜，而且 PM 可以直接参与。动用自我批判或 multi-agent 之前，请先确认自己真的优化过 prompt。

**第三，重要的是生产行为，而不是 benchmark 行为。**某项技术在 benchmark 上增加 25% 准确率，如果你的查询不像 benchmark 里的查询，在生产中可能就是 0%。我学会培养直觉的方法，是阅读真实用户轨迹：每周从生产里抽取二十个真实 session，阅读 agent 实际做了什么，在哪里摔倒，在哪里因为错误原因而成功。这是一项并不光鲜的工作，而且大多数内容不会出现在 dashboard 里。但它也是我做的杠杆最高的一件事。

**第四，是我逐渐称之为缓存纪律的东西。**这更像是工程问题而不是产品问题，但它有值得理解的产品含义。现代 LLM serving 系统会在前缀匹配时跨调用复用计算，也就是所谓的 KV-cache。高缓存命中率会大幅降低延迟和成本；低命中率会大幅提高两者。Manus 背后的团队对此发布过一篇非常出色的复盘，他们认为 KV-cache 命中率是 AI agent 最重要的生产指标。对 PM 来说，实际含义是：不断改变 prompt、动态修改工具定义，或者每一轮都重组 agent 的上下文，这些在产品层看似无害的事情，可能会通过破坏缓存，悄悄摧毁你的单位经济模型。修复方式是保持 prompt 前缀稳定，在扩展上下文时追加而不是重写，并把 prompt 结构当成一个修改起来确实有成本的东西。

**第五，衡量你的路由，而不只是衡量你的模型。**跟踪模型输出有多好很容易。跟踪你使用模型有多高效更难，也更重要。RouteLLM 论文里的三个指标是很好的起点：命中昂贵路径的请求比例、恢复的性能差距、调用-性能阈值。但底层观点比路由更广。你做出的每个架构决策都会在某个地方增加成本；问题是这些成本有没有买到任何东西。如果你说不清自己的 retry loop 或 multi-agent 设置到底在质量上恢复了什么，那你就是在为没有真正做事的架构付钱。像怀疑供应商营销一样怀疑自己的架构。

**第六，也是我最想花时间谈的一点：增加架构很少能修好模型问题。**

这是我看到 PM（包括我自己）最慢才学会的一课。当模型输出不够好时，本能反应是在它周围加结构。包一层 retry loop。加一个 critic。构建一个 multi-agent 系统。使用 tree search。每一种都是有真实用例的合法技术，但它们都不能替代模型本身具备完成任务的能力。

我的发现是，当底层模型能完成任务时，即便简单架构也能产出可靠结果。当底层模型不能完成任务时，增加架构只会产生一个以更微妙方式失败的复杂系统，而这比明显失败更糟，因为微妙失败不会在 QA 中被发现，最后会进入生产。一个比你需要的水平稍差一点的模型，套上自我批判循环，会产出看起来很详尽、但其实自信地错误的结果。一个能完成任务的模型，配上好的 prompt 调用一次，产出的就是正确结果。

所以我最终确定的操作顺序是：选择看起来有可能完成任务的最便宜技术，用真实数据测试它，只有在有证据表明复杂性正在解决真实问题时，才增加复杂性。复杂性应该跟随失败，而不是先于失败。

这是一种比"我们先设计 agent 架构吧"更慢、也没那么刺激的构建方式。但它会产出真正可用、也真正能赚钱的产品。

## 这把我带到哪里，以及下一步是什么

前沿模型每个季度都在变得更好、更便宜。AI 产品之间的差异不会来自你使用哪个模型。到明年，每个人都能以相近价格用上相同模型。差异会来自模型上方的那一层：你做路由有多审慎，工具设计得多好，prompt 结构有多清晰，以及你在复杂性值得时才增加复杂性这件事上有多自律。

那正是 AI PM 花时间的层。也是大多数成本决策发生的层，不管有没有人意识到。对创业公司来说，这些决策像招聘计划一样直接决定你的 runway。

**但还有一个显而易见的问题我还没有回答：你怎么知道一项技术是不是真的有效？**

这篇文章里的每条建议，"Chain-of-Thought 增加 10-30%"、"路由恢复 80% 的质量差距"、"如果有验证信号，自我批判带来 5-25%"，都隐含着你能在自己的产品上衡量这些东西。实践中，大多数团队做不到。他们发布一个 prompt 改动，然后相信感觉。他们替换模型，然后看客服工单。他们增加 retry loop，因为它感觉更安全。

这就是我接下来想写的内容。**"成本是产品决策"的配套篇，是"evals 也是产品决策"。**没有 evaluation system，上面的菜单就只是词汇表。你无法判断哪项技术有帮助，哪项技术在浪费钱，也无法知道一次"好的"模型变更是否悄悄破坏了 5% 用户的体验。eval system 会把菜单从理论变成你真的能用来导航的东西。

那就是下一篇。如果这篇让你有共鸣，下一篇会让它变得可操作。
