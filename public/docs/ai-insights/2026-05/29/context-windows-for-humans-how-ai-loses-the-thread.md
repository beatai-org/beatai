# 写给普通人的上下文窗口：AI 是怎么跟丢思路的

以及，为什么每一次 AI 对话都存在一个边界

![](https://cdn.jsdelivr.net/gh/beatai-org/beatai-assets@50756e864d1372dfaaa4100bc21289f6c418a4c0/ai-insights/2026-05/29/images/context-windows-for-humans-how-ai-loses-the-thread/01.jpg)
*图片来源：Waldemar Brandt, Unsplash*

类似的限制，作为人你早就感受过：

-   *谈话里信息同时砸过来太多，跟不上节奏*
-   *只看了一封长邮件最末一条回复，结果错过了开头那段最关键的信息*

每一种情况，底层都是同一件事：能同时保持在"鲜活状态"里的信息，就这么多。

## 摘要

上下文窗口，是一个 AI 模型在一次交互里能同时主动考虑的信息量。如果对话、文档或 prompt 拖得太长，部分信息可能被丢弃、被压缩，或者被分到更少的注意力。这就是为什么聊天机器人会忘掉早先的指示、偏离最初的话题、漏掉细节。

那上下文窗口到底是怎么工作的？为什么哪怕给它更多上下文，AI 还是会跟丢思路？接下来还是老规矩，用大白话讲清楚。

## 上下文窗口到底是什么

很多人跟 AI 说话时，下意识把它当成无限记忆的东西。其实不是。

大语言模型（LLM）跑在一个 [**上下文窗口**](https://platform.claude.com/docs/en/build-with-claude/context-windows) 里——这是它能同时主动考虑的文本量。包含你刚输入的 prompt、对话里更早的部分，以及上传文档之类的附加材料。

关键在于，这个窗口是有限的。它每次都在跟有限量的文本打交道；随着越来越多的文字填进来，对话里的某些部分就越来越难保留在主动视野里。

可以把上下文窗口想象成开车时的视野。前方看得清，但不可能像 Google maps 一样一眼看完整段路线。车一直在动，新东西不断进入视野，旧的就退到后面去了。

所以聊天机器人开头可能犀利、连贯，对话拖长之后状态就会涣散。早先那些信息倒不见得真"消失"了，只是不再完整地停留在模型的主动上下文里。

## 引擎盖下发生了什么

要真正理解上下文窗口，得先聊聊 [token](https://developers.openai.com/api/docs/concepts)。

AI 模型衡量文本的单位不是页、不是段，甚至不是词。是 token——由 [tokenizer](https://platform.openai.com/tokenizer) 切出来的小块文本。有时一个短词就是一个 token。有时一个长词会被切成好几段。标点和格式有时候也算。所以词到 token 没有完美的换算，但有个常用的英文经验值——一个 token 大概是 3/4 个词，平均每个词约 1.3 个 token。

为什么这事重要？因为语言模型不是把你的 prompt 当作一大块意义来处理的，它处理的是一串 token 序列。

第一步，把你的文本切成 token。第二步，把这些 token 转成模型能处理的数值表示。第三步，模型处理这条 token 序列，并通过一种叫做 [attention](https://developers.google.com/machine-learning/glossary#attention) 的机制，判断在生成下一个 token 时，序列里更早出现的哪些部分最相关。

核心限制在于，这一切只发生在当前上下文窗口里的那些 token 上。换句话说，上下文窗口就是模型在同一时刻能主动操作的 token 数量上限。

而这个 token 预算很快就会被吃光。所有输入和输出，包括模型自己的回复，都要占 token。所有内容都在同一个窗口里抢空间。

![](https://cdn.jsdelivr.net/gh/beatai-org/beatai-assets@50756e864d1372dfaaa4100bc21289f6c418a4c0/ai-insights/2026-05/29/images/context-windows-for-humans-how-ai-loses-the-thread/02.webp)
*作者绘制的 R 图*

所以如果说上下文窗口是一块白板，那 token 就是写在上面的字。写得越多，能同时让所有内容都保持可见的空间就越少。

## 把它想象成一块白板

当过老师或学生的人，都熟悉白板。一开始很容易跟上老师写的内容。可讨论一路进行，板上写得越来越多，最后就开始挤了。

到那时候，要把每件事都跟住就难了。旧的细节也许还在板上某处，但已经不那么容易被注意到、被用上。新内容开始抢地盘，整体的思路也就糊了。

上下文窗口里发生的事，差不多就是这样。模型每次只跟有限的文本打交道，这块空间一旦填满，要让每条早先的指令保持同等活跃，就更难了。

问题在于，主动工作区越来越难管理；一旦如此，对话里的某些部分就自然而然地退到了背景里——其中可能就包括关键部分。

## 今天的上下文窗口有多大？

下一个问题自然而然冒出来：白板到底多大？

答案是，比几年前大多了，但仍然有限。

现代语言模型能处理非常大的上下文窗口。OpenAI 当前的[模型文档](https://developers.openai.com/api/docs/models/compare)列出的上下文窗口，比如 GPT-5.4 mini 是 40 万 token，GPT-5.4 和 GPT-5.4 pro 是 105 万 token。Anthropic [写明](https://platform.claude.com/docs/en/build-with-claude/context-windows)许多 Claude 模型的标准上下文是 20 万 token，也宣布了 Claude [Opus 4.6](https://www.anthropic.com/news/claude-opus-4-6) 和 [Sonnet 4.6](https://www.anthropic.com/news/claude-sonnet-4-6?utm_source=chatgpt.com) 的百万 token 上下文。

听起来巨大，确实巨大。这么大的上下文窗口，能装下长对话、大文档，或者很长的一段代码。但它仍然不是无限记忆。而且，把这个上限做大，不是免费的。

在标准 [Transformer](https://arxiv.org/abs/1706.03762) 架构里，每个 token 都可以 attend 到序列里的其它 token，意味着 attention 的开销随着序列变长呈平方增长。

说人话：把上下文做大很多倍，就是在制造多得多的 token 与 token 之间的关系让它去考虑。这就是为什么长上下文 AI 一直是个工程难题，而不是"把窗口做大点"那么简单。

哪怕模型能吞下海量文本，你也不能往 prompt 里一股脑塞越来越多东西，还指望全部完美工作。

所以，更大的上下文窗口固然强，但有代价。空间多有好处，但要花算力、花钱、花工程力。意思就是，哪怕走到了百万 token 模型的世界，上下文仍然得管理，不只是扩张。

## 窗口被填满之后会怎样

也许你一开头就告诉模型：语气要正式、不要用项目符号、举例都从医疗行业出。二十条消息之后，它突然开始转向零售业的例子。模型没有像人忘掉生日那样"忘记"什么。只是那些早先的指示，不再被以同样的清晰度和权重看待。

一个简单的想象方式是：假设模型一次只能主动处理 100 个 token。只要对话不超出这个限度，所有内容都在视野里。可一旦第 101 个 token 来了，就必须让点东西出去。

早先的部分上下文从主动窗口里掉出去，腾地方给新内容。实际系统里的处理方式各有不同。有的可能直接丢掉最老的 token。有的会把更早的内容做个总结，或者把系统指令单独保留。也有应用自己来决定哪些消息留在 prompt 里、哪些剔掉。

所以对话越长，模型越倾向于优先抓住那些最近的、被重复过的、或者最容易抓上手的内容。

## 上下文窗口 vs. 记忆

人们听到"记忆"两个字，常常会想到内存、硬盘或者数据库——存起来的信息，需要时再取出来。

上下文窗口不一样。它不是一个存储系统。它是模型在这次交互里能主动处理的那一片文本。

这个区别很关键，因为它改变了你对模型在做什么的理解。聊天机器人不是在搜索一个稳定的归档库——你说过的所有话——按需把细节拎回来。它处理的就是此刻窗口里那些文本。某个细节如果不在当前主动上下文里，模型不能仅仅因为它早先出现过，就可靠地拿来用。

这就是为什么模型可以在某一刻完美执行一条指令，下一刻又像是把它跟丢了。问题不在于这条信息从某块硬盘上被删掉了。问题在于，它不再属于模型当前主动的工作集。

所以把上下文窗口叫做"记忆"，会让人误解。计算机内存意味着存储和检索。上下文窗口更接近一块临时工作台。

聊天机器人看起来在"忘事"时，信息不见得被"擦掉"了。只是不在那部分模型当前能主动用上的对话里了。

## 怎么顺着上下文窗口干活，而不是逆着它

理解了上下文窗口，很多让人抓狂的 AI 行为，都开始说得通了。

知道几个技巧，缓解起来也容易得多。

第一条策略：让重要指示保持可见。真正关键的东西——语气、格式、约束、受众、要用的例子、要避开的地雷——说清楚，必要时再说一遍。不要假定二十条消息之前的某条指令会一直有效。

也要想一想重要信息出现在哪儿。一项叫 [*Lost in the Middle*](https://aclanthology.org/2024.tacl-1.9/) 的著名研究发现，语言模型在用相关信息时，如果信息出现在上下文的开头或结尾，往往用得更好；如果同样的信息埋在中间，用得就差。换句话说，位置很关键。

如果你希望模型可靠地用上某个东西，更好的做法通常是把它放在 prompt 靠前的位置，或者在末尾再重申一次，而不是塞在一大段文字的中间。

第二条策略：把大任务拆成小任务。与其要求模型同时把所有东西摆在主动视野里，不如把工作切成几块。这样能缓解拥挤，模型也更容易保持对齐。

第三条策略：边走边总结。长对话里，一个简短的 checkpoint 能帮整段对话重新校准——目标是什么、约束有哪些、目前已经定了什么。这给模型一份重要上下文的清洁版，而不是逼它在一团之前所有信息的乱麻里来回腾挪。

第四条策略：把最重要的材料放在离当下任务最近的位置。把任何重要的东西重新带回当前 prompt 里，比指望系统自己回头去给它加权重要管用。

最后，记住更多上下文有帮助，但不是魔法。更大的上下文窗口让模型有更多操作空间，但并不保证完美的注意力、优先级或理解。文本更多不总是更好——有时候只是更多干扰。

换句话说，最聪明的用 AI 方式，是帮它把对的东西稳稳地抓住。
