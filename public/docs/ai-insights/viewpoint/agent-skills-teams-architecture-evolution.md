# Agent/Skills/Teams 架构演进过程及技术选型之道

## 背景

> 辣评：上来先争论“要不要多智能体”，却不先画清问题边界，这和我当年在 MIT 机器人赛场上先选喷漆配色再写控制程序一样不靠谱。

![图片1](/docs/ai-insights/viewpoint/images/agent-skills-teams-architecture-evolution-01.jpg)

从生成式LLM爆发到Agent狂飙，行业几乎一直在“边开飞机边换引擎”，顺便还在机舱里改线路。过去近半年，Anthropic围绕Claude Code连续落地Agent Skills、Agent Teams等范式，直接把Agent的构建逻辑和能力边界重新洗牌。站在这个时间点，再拿旧框架讨论“如何做优秀Agent”和“如何选架构”，很容易像拿十年前导航闯新城区，嘴上说稳，方向盘却越打越偏。

在聊具体选型前，先把演化路径画清楚，这才像工程讨论而不是“谁嗓门大谁有理”。这条路径并不神秘：从提示词单点调用、工作流编排，到多智能体协同、自主规划，再到Agent Skills复用与Agent Teams并行探索。路径看清，选型才不会变成“会议室掷骰子”。

下面这份分析不是“PPT体悟”，而是基于实际落地经验，并与下面几篇文章交叉校准，集中讨论Agent、Multi-Agent、Agent Skills、Agent Teams的架构演进与选型逻辑：

- 《Towards a Science of Scaling Agent Systems》 By Google Deepmind[1]
- 《The Complete Guide to Building Skills for Claude》 By Anthropic[2]
- 《Orchestrate Teams of Claude Code Sessions》 By Anthropic[3]

## Agent架构的演化本质

> 辣评：把术语壳子都拆掉后，底层逻辑很朴素：你一直在补两件事，知识注入和记忆管理。其余大多只是包装层厚薄不同。

为什么市场上的Agent架构看起来像“工具链大爆炸”？根子并不神秘，也不只是炫技，而是给大模型短板打补丁。说穿了，基础模型暂时还做不到完美内化领域知识，也做不到低成本复用长期记忆，于是工程团队只能外挂能力。Agent演化史，本质上就是一部“知识注入+记忆管理”的补偿史，主线清晰得像实验室白板上的红色箭头。

![图片2](/docs/ai-insights/viewpoint/images/agent-skills-teams-architecture-evolution-02.jpg)

不妨做个“理想宇宙”设想：如果某天LLM天生就具备完美的领域知识注入和自主记忆，你把海量行业文档、业务规则一股脑喂进去，它就能瞬间记住并稳定执行任务，那今天讨论的RAG、Multi-Agent、Workflow、Skills很可能会集体失业，连离职报告都不用写。因为模型已经在底层把“学什么、记什么”一次性解决。

但现实没有这么体面。回看2023~2024年，行业早期普遍把“再训练模型”当作垂类知识注入的正解：延续BERT时代的“预训练-微调”范式，在基础模型上叠SFT、DPO，再加RLHF、GRPO，试图把领域知识直接刻进参数。包括基于Qwen早期版本做多轮深度训练和微调的实践，也把这条路的成本和边界照得很清楚，像把账单摊在强光下逐行对账。

但是，随着训练过程的深入，几个难以回避的痛点逐渐浮出水面：

- 训练成本高昂且周期长：每一次针对垂直领域的训练，都需要投入巨大的人力物力去清洗数据、构造合成数据、设计评测集。这不仅需要昂贵的GPU算力资源，更伴随着漫长的训练时间周期。
- 效果评测与泛化难题：训练完成后，如何科学证明新模型相较基座模型有显著提升，同时又不牺牲通用泛化能力，是巨大挑战。很多团队在提升特定任务表现时，意外触发其他场景的“灾难性遗忘”，结果是垂类任务看似更强，通用能力却明显回落。
- 基座迭代速度远超训练周期：这是最致命的一点。开源或闭源基座都在非线性高速迭代。团队往往耗费数月、投入大量资源训练出专属模型，新一代基座却已发布，原生能力可能直接超越旧版本训练成果。这种“刚毕业就失业”的局面，让单纯依赖训练构建领域模型越来越不划算。

除了成本和时效，硬件门槛与模型生态变化也在同步加压。Scaling Law继续生效，顶尖模型参数量越滚越大，单机和小集群越来越扛不住训练账单。更现实的是，最强模型很多是闭源；即使用开源顶尖模型再训练，效果也常追不上闭源最新基座。在这种投入产出比失衡的盘面里，继续死磕训练，很像拿航天级燃料点灶台火，响动很大，位移很小。

这就把问题钉死了：现阶段LLM在特定领域知识内化与长周期记忆管理上仍有明显缺口。既然“向内改参数”要么走不通、要么性价比过低，工程团队自然会转向“向外做系统”，在不改模型权重前提下，用架构把知识注入做得更准更快，像给大脑配一个靠谱的外接存储而不是强行开颅扩容。

这也是Agent架构演化的起跑线：必须在模型外围搭结构和工具，用工程化机制完成知识检索、上下文组装和记忆维护。与其逼模型“记住一切”，不如让模型“随时拿到并理解所需知识”。顺着这条思路，演化才逐步分出四条主路径：“Single Agent → Multi-Agent → Agent Skills → Agent Teams”，像从单线程一路升级到有调度器的并发系统。

## Single Agent：知识注入与上下文窗口博弈

> 辣评：单智能体不是“低配版”，而是基线组。跳过基线直接堆并行协作，像没做地面测试就把火箭推上发射架。

在Agent落地早期，Single Agent（单智能体）几乎永远是第一站。逻辑很朴素：模型不能直接内化特定领域知识，那就先把关键知识压进System Prompt，先让系统“能答、答得动、答得快”，先把自行车骑稳再谈涡轮增压。

这种做法的优点非常“工程友好”：实现成本低、开发速度快。你把领域知识整理好，配上清晰指令写进System Prompt，再用基础模型原生ReAct模式去调用工具、记录上下文、推进任务。对代码片段生成、文案写作、标准化输出等场景，它常常是ROI最高的起步方案，像先用扳手把关键螺丝拧紧，而不是直接订一整套自动化产线。

但实践很快暴露一个致命瓶颈：Context Window会爆炸。

尽管主流模型都在宣传百万甚至千万级Token上下文，但生产里把海量文档直接塞给模型，效果常常并不理想。核心技术事实很残酷：长上下文不等于长记忆。输入一旦过阈值，模型容易出现“Lost in the Middle”，关键细节被注意力稀释，最后像在一堆会议纪要里把最关键那页落在打印机上。

这里把定义先钉住：本文的Single Agent主要指“狭义”ReAct自主Agent，即System Prompt驱动、串行调用工具的原生模式。至于多分支判断的复杂Workflow，更像高级Tool或Sub-Agent，本节不展开，先把边界线画直，省得术语打群架。

一句话总结，单Agent的优劣非常鲜明：

- 优势：架构原生、开发链路最短、运行效率高，适合快速构建Demo或处理知识依赖较少的场景。
- 劣势：高度依赖上下文窗口质量与长度。一旦涉及大量领域知识注入，极易触发上下文爆炸，导致模型注意力分散，稳定性下降。

关键问题立刻出现：当单点能力撞上上下文天花板，怎么在保持灵活性的同时承载更多知识，而不是把提示词越堆越像超市小票？

面对这个瓶颈，行业最常见的补救动作是引入RAG（检索增强生成）。

RAG可以看作Single Agent的一次实用升级，核心是“先搜后答”：先用检索工具做召回，只把与问题最相关的片段塞进上下文，再让Agent回答，相当于先把资料室里最相关的文件夹拎出来再开会。

这确实在一定程度上绕开了Context Window硬限制，让Agent按需进食而不是全量硬吞。但它有一条硬依赖链：Garbage In, Garbage Out。检索没找准，后端模型再强也会在错误语料上一本正经地跑偏，仿佛穿着西装自信地走错考场。

这里还有一道常被低估的能力断层。RAG前置检索通常依赖关键词匹配（如BM25）或小参数Embedding模型（如BERT、BGE等）。即便近年LLM Embedding在进步，整体语义理解与推理深度仍常弱于“大模型直接阅读全文”。这就容易出现漏召、误召，把关键证据挡在门外，留下边角料在屋里高谈阔论。

基于以上分析，单Agent的适用边界可以画得更清楚。它不是万能，但在以下条件下依旧是性价比最高、落地最快的选择，属于“便宜、好用、还能准时交付”的朴素工程解：

1. 场景复杂度较低：业务逻辑相对简单，不需要复杂多步推理或长链条规划。
2. 知识体量可控：领域知识总量适中，或经清洗后核心指令与背景知识能在2万Token内表述清楚，可直接通过System Prompt注入。
3. 检索质量有保障：当必须使用RAG时，前提是知识库结构清晰，且现有检索算法（关键词或向量）具备较高召回准确率。

简而言之：需求“小而美”，或领域边界清晰、检索链路成熟时，单Agent完全够用，没必要为了“看起来高级”过度设计。可一旦进入海量非结构化数据、复杂推理、或检索准确率极敏感场景，就该升级架构，不要硬扛，更别把“将就能跑”误判成“长期可用”。

## Multi-Agent：架构隔离与通信带宽的权衡

> 辣评：多智能体最贵的不是并行，而是协调。只算吞吐不算通信损耗，就像只算火箭推力不算姿态控制。

面对单Agent在海量知识注入和复杂场景上的天花板，Multi-Agent（多智能体）自然登场。这不是简单“多开几个Agent窗口”，而是组织方式升级。以阿里云客户服务领域云小二Aivis实践为例，通过Planner（规划者）、Reasoner（推理者）、Executor（执行者）协同，把宏观问题拆成微观子任务并分工处理，像把一场复杂抢修从“单人硬扛”变成“专业小队协作”。

Multi-Agent形态很多，Google论文主要归为四类：Independent、Decentralized、Centralized、Hybrid。

![图片4](/docs/ai-insights/viewpoint/images/agent-skills-teams-architecture-evolution-04.jpg)

- Independent：多个Agent并行处理子任务而不沟通，只在最后汇总结果。
- Decentralized：点对点网状结构，Agent之间直接沟通共享信息并达成共识。
- Centralized：中心辐射模型，由中央Orchestrator分配任务并综合输出。
- Hybrid：结合层级监督和点对点协调，平衡中央Orchestrator控制与执行灵活性。

前两种可看作“只有Sub-Agent”的形态，后两种都引入中央Orchestrator主Agent。核心逻辑是路由分发与领域隔离，本质上是“谁来拍板、谁来干活、谁来兜底”：

- 主Agent（Orchestrator）：扮演“大脑”，负责意图识别和任务路由，判断“问题该交给谁”，不必背负全部领域知识。
- 子Agent（Sub-Agent）：拥有独立Identity空间，内化特定领域专业知识（如ECS远程诊断、RDS性能优化）。每个子Agent只专注一类垂直场景，Prompt更精简、知识更聚焦。

这套设计确实带来两类明显收益，账面上看起来很诱人：

- 降低单体复杂度：把庞大领域知识打散，降低单个Agent Context Window爆炸风险。
- 独立调优：子Agent可独立迭代。若某个子Agent效果不佳，仅需局部优化，不影响其他模块，维护灵活性更高。

但当Agent数量上来，比如一个Orchestrator要调度上百个Agent，新瓶颈会很快冒头。Multi-Agent不是银弹，它更像把一个大难题拆成一堆小难题再加一个总难题：

- 路由准确率压力：当Sub-Agent扩到几十上百，主Agent面临巨大分类决策压力，需要在极短上下文内准确分发。一旦Misrouting，后续努力全线偏航，风险会随节点数量持续叠加。
- 局部最优导致上下文割裂：这是Multi-Agent最隐蔽也最致命的问题。子Agent常只关注自身局部最优路径，缺乏全局上下文和完整用户意图感知，容易出现重复执行与结论冲突。

- 重复执行：例如用户问“ECS远程无法连接”，Agent A已诊断“资源负载高”；用户追问“为何负载高”，Agent B接手后不知前文，可能重复做同样检测，浪费算力并增加延迟。
- 结论冲突：不同Agent基于局部信息得出的结论可能与前文矛盾，导致回答逻辑不自洽，用户体验显著下降。

为缓解上下文割裂，你当然可以让大家共享Context历史；但工程里很快会撞上通信带宽上限，像群聊消息一多就没人看完整楼层：

- 信息有损压缩：主Agent传给子Agent的常是Summary/Rewrite后的上下文而非原始对话流，有损传输可能丢关键细节。
- Token爆炸与耗时增长：若强行扩大通信带宽以传递更多上下文，会迅速引发新的Context Window爆炸，并显著拉长LLM生成时间和整体链路耗时。

所以，Multi-Agent确实解了知识隔离，却把复杂度挪到了通信带宽与协同治理上。想稳定跑出效果，就得持续投入人力打磨节点、协议、摘要策略和边界Case。典型边际效应递减：节点越多，稳定性保障难度非线性上升，收益却越来越依赖人工“看护”，最后团队像在托儿所里同时盯一百个会写代码的小孩。

因此Multi-Agent是一把双刃剑：它能靠分工突破单点上限，也会引入协同损耗。如何在“架构隔离”和“通信带宽”间拿到平衡，才是多智能体工程成败线，不是“Agent数量越多越先进”的面子工程。

## Agent Skills：可复用与渐进式的能力披露

> 辣评：Skills最实在的价值，是把“多人群聊式协作”拉回“单体可控执行”。能一本手册解决的事，没必要拉三十个人开会。

面对Multi-Agent里的通信损耗、路由误判和维护压力，越来越多团队开始找更稳妥路径。Anthropic在《The Complete Guide to Building Skills for Claude》中给出关键方向：与其盲目堆Multi-Agent，不如把能力沉淀成基于文件系统的可复用包，也就是Agent Skills，先把“组织内耗”换成“文档驱动”。

这次转向背后的逻辑很硬核：引入Multi-Agent本来是为知识隔离和高效注入，代价却是上下文管理与通信机制复杂化。如果能在不牺牲上下文稳定性的前提下动态加载知识，那么重型多智能体通信就不该当默认配置，毕竟不是每个问题都值得启动“星际舰队模式”。

Agent Skills本质上回归Single Agent架构本体，但赋予其强动态扩展能力：

- 能力封装复用：将复杂领域知识、操作规范、最佳实践封装为独立Skills文件包（类似操作手册），在不同Agent中快速复用。
- 按需调度：主Agent不再预加载全部知识，而是在运行中按任务需求动态读取对应Skills文件。
- 渐进式披露（Progressive Disclosure）：Agent先通过目录概览定位所需技能，再逐步深入执行细节；执行中若发现信息缺口，可继续触发后续Skills补全。

这套模式让单个Agent拿到“局部专业化”能力：宏观上保持统一记忆和状态，微观上又能像调用工具一样灵活拉起垂直领域知识，既有总控台，也有随手可取的操作手册。

很多人会问：“这不就是动态改System Prompt吗？之前翻车的坑不还在吗？”

关键差异在认知一致性。早期方案常直接动态替换System Prompt，容易触发认知冲突（Cognitive Dissonance）：系统指令从A切到B，历史对话却按A生成。模型会陷入“现在按B、之前按啥”的错位，结果就是逻辑抖动甚至幻觉，像考试中途突然换题纲还不允许翻草稿。

Agent Skills巧妙绕开了这个坑：System Prompt保持恒定，核心人设与基础约束不变，认知底座稳定；变化的是User Prompt层，Skills内容以用户输入或工具返回结果渐进注入。对模型而言这是“新增资料”，不是“强制换脑”，稳定性自然更好，像给工程师补说明书而不是现场重装人格。

所以Agent Skills通常能拿到三类收益，而且每一类都很“省人省心”：

- 低成本知识注入：将海量知识“说明书化”，模型按需阅读，无需全量预加载，比Multi-Agent更轻，也常比RAG更精准。
- 全局上下文一致性：始终由同一主Agent执行，完整掌握已执行步骤、已读取Skills与当前任务状态，显著减少信息割裂和重复劳动。
- 规避Context爆炸：通过“读一点、做一点、再读一点”的流式处理，控制瞬时上下文长度。

当然，Skills也不是魔法。切换过频时，累计上下文一样会变长。实际落地通常要配合上下文压缩或滑动窗口策略，及时清理无效中间信息，避免系统把带宽浪费在历史噪声上，不然就会变成“知识管理系统很先进，真正关键那行没人在看”。

从Multi-Agent的“分而治之”到Agent Skills的“聚而用之”，这条演进很有工程美感：用文件系统的结构化能力替代复杂通信协议，用渐进式披露替代暴力全量注入。对追求高稳定、低维护、还要处理海量领域知识的企业场景，它往往是更稳的当下解，也更像能长期值夜班的系统，而不是首周惊艳、次周告警爆炸的样板。

## Agent Teams：“协同共创”的探索式形态

> 辣评：Agent Teams适合探索未知，不适合当KPI装饰。没有中心化校验的并行试错，通常只会更贵、更快地犯同一种错。

在Agent架构前沿，Anthropic在实验性文章《Orchestrate Teams of Claude Code Sessions》中提出Agent Teams。它和Multi-Agent里的Independent/Decentralized有相似基因，但目标不一样：它瞄准的是复杂且未知的问题空间，更像“探索队”而不是“流水线班组”。

要理解Agent Teams的价值，先把它和传统Multi-Agent拆开看，别把两者都统称“多开几个AI”：

- 传统Multi-Agent：Sub-Agent更像独立员工，接收指令后独立完成任务，再向主模型提交结果。很多场景下子Agent彼此不交流，上下文隔离，无法利用彼此中间过程（当然也可设计成点对点通信）。
- Agent Teams模式：Agent被组织成真正“协同小队”，具备并行探索、共享上下文、动态协同和目标一致四个特征。

- 并行探索：多个不同Identity的Agent同时启动，从不同角度并发求解。
- 上下文共享：所有队员在共同Task List或Shared Context中实时写入进度、发现和思路。
- 动态协同：Agent不仅知道自己在做什么，也能看到队友进展，打破信息孤岛。
- 目标一致：所有Agent共享同一终极目标，只在过程上分工。

Agent Teams并不是为“领域知识注入”或“Context Window爆炸”而设计，它的初衷是解决高不确定性决策问题，也就是“地图上压根没标这条路”那类任务。

当你面对一个没有标准答案、甚至起手路径都不明确的问题时，串行单线推进就像蒙眼走迷宫：

- 单一路径风险：传统单Agent或串行Multi-Agent通常沿一条高概率路径推进，方向错了就整体失效。
- 多维试错能力：Agent Teams可动态发起多个子身份并行尝试不同解法（如代码修复、配置检查、日志分析）。
- 最优解涌现：并行跑通多条路径后，可比较中间结果，汇聚最优方案，或融合多个方案优点。

Agent Teams代表一种新的工程哲学：在未知区域里，并行多样性常常优于串行确定性。它更适合复杂研发调试、开放式创意生成、多因素耦合故障根因分析这类“没有现成路线图”的任务，核心价值是用多条试探路径换掉单点误判。

当然，这套模式有明确成本：并行减少等待，但算力账单会成倍增加；共享Task List若设计不当，也会引发冲突或死循环。并且Agent Teams并非全程并行，主Agent仍要拆任务、判定哪些能并行、哪些必须串行依赖。即便如此，并行探索+上下文共享带来的能力跃迁，依然非常可观，前提是你先把“协作交通规则”写清楚。

## 如何科学地构建Agent系统

> 辣评：这部分最有价值的变化，是从“我觉得”转到“我验证过”。工程世界里，证据永远比气势更值钱。

在Agent落地浪潮里，早期大量架构设计确实靠直觉和经验推进。一是演化太快，团队很难慢下来做系统分析；二是LLM系统相比传统软件不确定性更高。但只靠“感觉对了”终究不够工程化，如何科学构建Agent系统，仍是工业界和学术界共同要啃的硬骨头，毕竟“凭手感”不适合作为长期运维策略。

Google DeepMind论文《Towards a Science of Scaling Agent Systems》（可译“迈向智能体系统的科学”）给出了基于大规模Benchmark的实证方法。虽然论文写作时尚未覆盖Anthropic后续的Agent Skills和Agent Teams，但它对Single Agent与Multi-Agent的结论仍有很强参考价值，至少比“我同事觉得这样更稳”更可复现。

论文对比了Single Agent与四种Multi-Agent（Independent、Decentralized、Centralized、Hybrid）共五类主流架构，产出了一批“反直觉但好用”的结论。结合前文，可直接拿来校准选型策略，防止团队在架构会上反复原地转圈。

### 1. 模型越强效果越好，但并非Agent越多效果就越好

![图片5](/docs/ai-insights/viewpoint/images/agent-skills-teams-architecture-evolution-05.jpg)

Google在OpenAI GPT、Google Gemini、Anthropic Claude上评估Single Agent与四类Multi-Agent，结果显示模型能力与架构效果关系并不线性。通常模型越强，Agent效果越好，二者总体正相关，但“强模型+错架构”照样会翻车。

但Multi-Agent绝不是万能按钮。有时增益显著，有时反而降效。换句话说，不是把Agent数量堆上去、把架构名词换成“多智能体”，结果就会自动变好；系统不会因为名字更酷就自己提分。

### 2. 尽量降低沟通成本和通信带宽

Google实验给出一个很实在的信号：在固定Token预算下，Agent间高频沟通会显著拉低整体效果，话聊太满，推理就会被挤到角落。

- 原因：沟通会消耗宝贵Context Window，挤占推理与知识注入空间。
- 启示：能由单个Agent内部消化的逻辑，就不要拆成多轮通信。除非必要，应优先低通信带宽甚至零通信架构。

### 3. 单Agent的45%阈值法则

实验数据显示，当单个Agent任务成功率达到45%以上时，继续加Agent的收益就开始明显递减，甚至转负，这条阈值像在提醒你“先把基线练到位，再谈花活阵型”。

- 核心价值：盲目增加Multi-Agent数量并不一定提升上限。
- 警示：若单智能体基线已超过45%，继续叠加复杂协同机制可能反而拖累整体表现，应优先简化架构。

### 4. 错误放大效应

最值得警惕的数据是：纯Independent架构可能把错误放大17.2倍；引入中心化机制后，错误放大可压到4.4倍。这个差距已经不是“有点风险”，而是“没有总控就可能集体跑偏”。

![图片6](/docs/ai-insights/viewpoint/images/agent-skills-teams-architecture-evolution-06.jpg)

- 解读：缺少监管的“群策群力”很容易滑向“集体幻觉”。没有强Manager做校验纠偏，多个Agent一起犯错并互相背书的概率会迅速上升，最后像一群人同时拍胸脯说“我查过了没问题”。
- 结论：用Agent Teams做并行探索时，必须配套强中心化决策与回滚机制，否则稳定性很难交付给生产环境。

### 5. 场景决定架构：没有万能钥匙

Google在不同任务Benchmark上跑多种架构后得出明确结论：任务类型决定最优架构，没有全场通吃的答案，别指望一把瑞士军刀解决所有手术。

![图片7](/docs/ai-insights/viewpoint/images/agent-skills-teams-architecture-evolution-07.jpg)

- 规划类任务（PlanCraft）：逻辑链强、工具依赖少，Single Agent通常最高效，避免不必要工具与子Agent调度开销。
- 工具使用类任务（WorkBench / BrowseComp-Plus）：涉及工具规划、选择与浏览器检索，天然更适合去中心化Multi-Agent。
- 垂类场景任务（Finance Agent）：错误零容忍，中心化协作效果最佳，可在保留并行度同时严格把控每步操作，平衡效率与安全。

## Agent架构选型之道

> 辣评：从P0到P3这套顺序很务实：先用最小复杂度拿到可行解，再用复杂度预算换上限。这就是工程团队该有的克制。

回到这四条演进路径：“Single Agent → Multi-Agent → Agent Skills → Agent Teams”。它们不是互斥替代，而是面向不同复杂度场景的工具箱。落地时怎样选更稳？结合一线实践和Anthropic、Google公开研究，可以提炼出“由简入繁、按需升级”的方法论，先把简单问题用简单方法打穿。

![图片8](/docs/ai-insights/viewpoint/images/agent-skills-teams-architecture-evolution-08.jpg)

### 架构选型建议

理想建设路径应遵循奥卡姆剃刀：如无必要，勿增实体。翻译成工程话就是，先把简单方案打透，再为真实瓶颈付复杂度账单。选型优先级通常如下，谁都别一上来就点“顶配套餐”：

- P0：能用Single Agent解决的，绝不上复杂架构。
- P1：遇到知识瓶颈，优先引入Agent Skills，通过动态渐进加载扩展能力边界。
- P2：仅在上述方案失效且对效果上限有极致追求时，再谨慎启动Multi-Agent，并做好长期调优准备。
- P3：针对高度不确定的探索性任务，灵活叠加Agent Teams并行协作能力。

Agent架构没有绝对“最好”，只有“最合适”。按这条路径推进，通常能少走弯路，用更低成本做出更稳、更智能、可落地的企业级系统，也更容易把试点变成可持续生产。

## 总结

> 辣评：Less structure, more intelligence 不是墙上海报，而是工程约束。结构加得越晚越好，证据拿得越硬越好。

随着Agent技术成熟，建设方式正在从“凭感觉调参”转向“系统工程”。Google论文数据、Anthropic实践建议，以及云小二Aivis等一线经验，最终都指向同一原则：架构复杂度必须和问题复杂度匹配，不能拿重武器打小目标，也不能拿玩具锤子去修发动机。

Manus AI官网那句“Less structure, More intelligence.”（更少的结构，更多的智能）很适合作为收束。盲目追求Multi-Agent“高规格”会陷进通信泥潭和错误放大；该并行时死守单点Agent又会错失效率红利。只有基于场景特征，持续权衡复杂度、成本、错误控制和并行收益，才能做出真正健壮、高可用、可落地且更智能的Agent系统，这才是“少走神、少返工、少熬夜”的正经工程路线。
