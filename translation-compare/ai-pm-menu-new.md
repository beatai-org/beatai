# AI PM 的菜单：成本与质量权衡的实战指南

![一份给 AI PM 的菜单，用来更慎重地构建 AI 原生产品](./images/the-ai-pms-menu-a-field-guide-to-cost-quality-tradeoffs/01.png)
*一份给 AI PM 的菜单，用来更慎重地构建 AI 原生产品 :)*

我在创业。NomNow 是一款多模态 AI 营养师，现已上架 App Store。一路做下来，要说最想塞给年轻一点的自己读的那条经验，就是：**对一家基于 LLM 的初创公司而言，成本纪律不是优化，是生存。**

传统 SaaS 里，每个用户的边际成本接近零，松懈很久也无所谓。换成 LLM，边际成本就是架构设计阶段拍板那一刻定下的——而且不便宜。一款消费级 AI 应用，如果把每条消息都路由到最强的前沿模型，每位活跃用户每天能烧掉 0.50 到 2.00 美元。一旦放到有点规模的盘子里，这就是整个生意本身。

所以开始做 NomNow 时，在写下第一行 agent 代码之前就敲定了一件事：**路由图到底长什么样？**

诱惑当然是全都丢给 GPT-4o。多模态记录在它上面跑得漂亮——半夜拍的一张炸酱面照片、含糊不清的一段语音备忘、一句"今天和昨天吃的一样"，都能处理。但把简单活儿也送过去——"我达成目标啦！"这种消息、"豆腐里有多少蛋白质"这种问题、各种闲聊跑题——等于在用前沿模型的价码解决规则引擎就能搞定的问题。

![](./images/the-ai-pms-menu-a-field-guide-to-cost-quality-tradeoffs/02.png)
*左：鼓励类消息走规则引擎。右：多模态记录走 GPT-4o（当时的 SOTA）。*

于是做了路由。多模态记录走 GPT-4o（当时的 SOTA）；纯文本的营养问答走更便宜的模型，配合检索；鼓励类消息走规则引擎，零 token；跑题闲聊给一段温和的引导，也零 token。用户体验一样，平均成本大致低一个数量级。

![](./images/the-ai-pms-menu-a-field-guide-to-cost-quality-tradeoffs/03.png)

这一类决策——以及之后跟进的几十个同类决策——正是这篇文章要谈的。**对 AI 产品来说，成本不是 PMF 之后再回头修的东西。成本就是产品的一部分，必须从第一天就设计进去。**

## 为什么这是产品决策，不是工程决策

两个原因。

第一，不同实现之间的成本差距巨大。同一个功能，两种做法，token 成本可以差 30 倍甚至 50 倍。这已经不叫"优化"，这叫"做得起 vs 做不起"。事后再怎么打补丁也填不上这道沟。

第二，每一个成本旋钮同时也是产品旋钮。加一道自我批判式的重试？用户感受得到延迟。路由到更小的模型？口吻和准确率都会变。用固定工作流取代开放式 agent？产品**能做什么**都会变。这些不是后端决策，是穿着工程外衣的产品决策。

也因此，谁负责产品体验，谁就得负责成本的形状。默认"哪里都用最强模型"是一种决策。默认"工程同学会想办法"也是一种决策。按经验，两种通常都错。

下面这张图——本领域常见的一张图，最早由 Artificial Analysis 整理，被 [Zilliz 关于 RouteLLM 的文章](https://medium.com/@zilliz_learn/routellm-an-open-source-framework-for-navigating-cost-quality-trade-offs-in-llm-deployment-7c4ee2158835)收录——一帧讲清了局面。没有"最好的"模型，只有一条前沿曲线，工作就是判断产品的每一处该落在曲线的哪一点上。

## 单次调用

默认选项。一条 prompt，一条回答。如果任务能用一句话讲完、答案在一段话里放得下，就别花架子。"把它做成 agent"的冲动，正是很多产品开始变差的起点。

## 思维链

让模型在回答前一步步思考。代价是多花 20% 的 token，换来推理类任务 10–30% 的准确率提升。**这是菜单上唯一的免费午餐。** 任务里只要涉及链式逻辑，就该打开。注意：小模型（<70B）上可能起反效果——它们会生成听起来挺顺、其实和底层逻辑对不上的"推理"。先测，再信。

## 路由

先分类请求，再送进足以处理的最便宜路径。在 NomNow 上最依赖的就是这一招。学界也有同样直觉：UC Berkeley 2024 年的 RouteLLM 论文显示，即使一个简单的二分路由器，也能在成本降低 2.5 倍的同时保留强弱模型之间 80% 的质量差，或者在成本降低 3.6 倍的同时保留 50% 的差距。如果每个请求都送到最强的模型，等于在为平均请求用不上的能力多付钱。

路由有三个值得跟踪的指标，全部来自那篇论文：**走最贵路径的请求占比**（应该很小，而且要清楚为什么走那条），**性能差距弥合度**（路由器抓住了多少质量差距——满分 1.0，随机约 0.5），以及**调用-性能阈值**（给定一个质量目标，最少需要多少比例的昂贵调用？）。这些指标能告诉你路由是在干真活，还是只在徒增复杂度。

**路由的主要瓶颈：分桶有多准，路由就有多准。** 用户不会按预设分桶说话。他们会写"我吃了两个苹果那是多少糖？"（同时记录 + 查询）；写"今天好烂"（闲聊？记录？情绪？）；写菜谱请求、天气问题，偶尔还来一段 prompt 注入。生产级的路由系统需要四种兜底：一道**置信度阈值**，低于该值就反问澄清而不是硬猜；一套**多意图处理**机制，应对跨桶的查询（本质上是个小工作流）；一条**超范围路径**，礼貌说明做不到什么；以及一道**安全覆盖**，处理 prompt 注入和敏感话题。没有这四样，路由只能算 demo，算不上产品。再往深一层说：**当产品边界清晰时，路由才是对的模式。** 对 NomNow 来说，"帮用户记录并理解所吃的食物"是一个有界问题。对开放式助手而言，路由会崩——桶越分越多，分类器越来越烂，这种时候大概率应该转向 agent。

## 工作流

步骤是预先定义的，模型负责在每一步填活。整条流程能画在白板上。比 agent 更便宜、更可靠、更好调试。**Anthropic 公开过一条指引，深以为然：如果用工作流就能解决，那就用工作流。** 大多数本该是工作流的产品，被做成了 agent，仅仅因为 agent 显得更酷。而这些 agent 都更差。

## Agent 循环

每一步由模型决定接下来做什么。流程没法事先画到白板上。任务真正开放时威力很大，反之则贵且不可预测。成本方差才是杀手：同一个查询，周二可能比周一贵 5 倍，只因为模型决定多试几下。对初创公司而言，这种方差既是财务问题，更是产品问题。

## 解耦式规划（ReWOO 一类）

agent 的一种变体：先把整条工具调用序列规划好，再把工具一次执行完（通常并行），最后合成。*当规划可预测时*，比标准 agent 循环大约便宜 5 倍。在结构化调研里效果好（"对比这三家公司过去四个季度"），一旦每一步结果都决定下一步，就崩。

## 自我批判与重试

模型先生成 → 另一次调用做批判 → 再一次调用做修订。如此循环。成本 6–10 倍，准确率提升 5–25%——**前提是有一个客观的"对"信号**（测试通过、规则引擎校验通过、引用核对通过）。没有这种信号，批判者就是在瞎猜，越改越偏的可能比改对还大。另外：延迟 6–10 倍，绝对不要塞进同步聊天的 UX。

## 树搜索（Tree of Thoughts、LATS）

每一步生成多个候选、打分、在树上搜索。在窄而可验证的任务上（数学、有测试的代码）提升惊人。成本 10–30 倍。在消费级产品里没用过，谁说自己用过都会让人怀疑。这套是给异步、高价值、强可验证的任务准备的。

![](./images/the-ai-pms-menu-a-field-guide-to-cost-quality-tradeoffs/07.png)
*图：Tree of Thoughts (ToT)，来源*

## 多 agent

由一个编排器协调多个专门化模型。Anthropic 公布的数据是：在调研任务上成本 15 倍，质量 +90%。合适的场景很窄：**真正并行、彼此独立的子任务。** Cognition（Devin 背后的团队）2025 年发过一篇《Don't Build Multi-Agents》，在编码场景下持相反观点——他们对编码的判断是对的，Anthropic 对调研的判断也是对的。动用多 agent 之前先问一句：*这些子任务真的能在不共享上下文的前提下并行完成吗？* 能就考虑用；如果它们需要不断对齐进展，最后得到的会是一个又贵又比单 agent 还不可靠的东西。

## 每周真正在想的事

光知道这份菜单还不够。更难的是养出从中挑选的判断力。在 NomNow 上一路做决策，反复回到几条原则，倒是顶得住事。

**第一，工作流几乎永远是正确的起点。** 不是说 agent 没用——很有用，NomNow 也有 agent 化的组件——而是把"需要 agent"这件事判断错的代价，远高于把"需要工作流"判断错的代价。一条工作流后来发现需要 agent 灵活性，可以扩展；一个 agent 后来发现是杀鸡用牛刀，就难往回退了——因为等回过神来，已经发布了一个单元经济跑不通、失败模式也没摸清的东西。Anthropic 给过一个粗略指引——0.10 美元/任务的预算能买到 30K–50K token，工作流绰绰有余，agent 就吃紧——这是个有用的直觉校验。如果产品一年要跑一百万个任务，本该是工作流却被设计成了 agent，等于无端烧掉七位数美元。

**第二，通常都有没用上的免费质量提升。** 思维链是最明显的一项——多花 20% 的 token，换推理任务两位数的准确率提升，这笔买卖每次都该做。还有别的。工具描述是被低估的一项：Anthropic 团队发表过数据，仅仅重写工具描述——同样的模型、同样的代码、同样的成本——就能把任务成功率拉高 5–20%。prompt 里的 few-shot 示例是另一项。一般规律是：prompt 层的质量提升，几乎总比架构层的质量提升更便宜，而且是 PM 能直接出力的环节。在伸手去抓自我批判或多 agent 之前，先确认 prompt 真的已经优化过。

**第三，重要的是生产环境的表现，不是基准上的表现。** 一项技术在基准上加 25% 的准确率，到了生产环境可能加 0%，只因为线上的查询长得不像基准里的查询。培养直觉的方法，是去读真实的用户 trace——每周从生产环境里拉二十段真实会话，看 agent 实际做了什么、在哪儿卡住、又在哪儿是"歪打正着"。这是不光鲜的活，绝大多数都不会出现在仪表盘上，却是手头杠杆最大的事。

**第四，渐渐想清楚的一件事，可以叫它缓存纪律。** 这更像是工程问题而不是产品问题，但有值得理解的产品含义。现代 LLM 服务系统会在前缀相同的调用之间复用计算——也就是所谓的 KV-cache。命中率高，延迟和成本就大幅下降；命中率低，两者就一起飙升。Manus 团队就此发过一篇精彩的复盘，主张 KV-cache 命中率是 AI agent 最重要的单项生产指标。对 PM 的实际启示是：频繁改 prompt、动态修改工具定义、每一轮都重新组织 agent 的上下文——这些事在产品层看上去人畜无害，却会悄悄破坏缓存、把单元经济拖垮。补救办法是：保持 prompt 前缀稳定，扩展上下文时用追加而不是重写，并且把 prompt 结构当成一种"改动有真实成本"的东西来看待。

**第五，要度量路由本身，而不只是度量模型。** 跟踪模型输出的好坏不难；更难、也更重要的，是跟踪使用得有多高效。RouteLLM 论文里那三个指标——走昂贵路径的请求占比、性能差距弥合度、调用-性能阈值——是不错的起点，但底层意思比路由本身更广。每一个架构决策都会在某处增加成本；问题是这笔成本到底买回了什么。如果说不清重试循环或多 agent 设置在质量上挽回了什么，那就是在为一套不干真活的架构付钱。看自家架构的眼光，要跟看供应商营销话术一样挑剔。

**第六，也是想多说两句的一条：加架构很少能修复模型问题。**

这条 PM（包括自己）学得最慢。当模型输出不够好时，本能反应是在外面再加一层结构。包一道重试循环；加一个批判者；搭一套多 agent 系统；上树搜索。每一项都是有正当用途的合法手段，但谁也代替不了模型本身能不能做这个任务。

观察下来的规律是：底层模型能完成任务时，再简单的架构也能产出可靠结果；底层模型完不成任务时，加架构得到的是一个以更隐蔽的方式失败的复杂系统——这比明显失败更糟，因为隐蔽的失败 QA 抓不到，会一路漏到生产。一个比需求略差的模型，包在自我批判循环里，输出会显得很周密，其实自信地错。一个能完成任务的模型，配上一段好 prompt 单次调用，输出干脆就是对的。

所以现在的操作顺序是：挑能合理完成任务的最便宜技术，在真实数据上测，等到有证据表明复杂度确实在解决真问题，再加复杂度。复杂度应该跟在失败后面，而不是抢在前面。

这种做法比"先把 agent 架构设计出来"慢得多，也没那么激动人心。但产出的产品是真能跑、真能赚钱的。

## 这条路把我带到哪里——以及接下来要写什么

前沿模型每个季度都在变得更好、更便宜。AI 产品之间的差异化，未来不会来自用了哪个模型——到明年，谁都能以相近的价格用到相同的模型。差异化会来自模型之上的那一层：路由设计得有多用心、工具打磨得有多到位、prompt 结构有多干净、加复杂度的纪律有多严——只在赚得回来时才加。

那一层正是 AI PM 花时间的地方，也是大多数成本决策真正发生的地方——不管当事人是否意识到。对一家初创公司而言，这些决策对 runway 的影响，跟招聘计划一样直接。

**不过有个明显的问题还没答：怎么知道一项技术到底有没有起作用？**

这篇文章里每一条建议——"思维链加 10–30%"、"路由弥合 80% 的质量差距"、"有校验信号时自我批判加 5–25%"——都隐含一个前提：能在自己的产品上量出来。实际情况是，多数团队量不出来。改一版 prompt，全凭感觉；换一个模型，就盯客服工单；加一道重试循环，就因为这样"看着保险"。

这就是下一篇要写的事。**"成本是产品决策"的姊妹篇，是"评测也是产品决策"。** 没有评测体系，这份菜单不过是一堆词汇——既说不清哪项技术真在帮忙、哪项在烧钱，也看不到一次"看上去更好"的模型改动是不是悄悄把 5% 的用户体验弄崩了。评测体系是把这份菜单从理论变成真正可导航的东西。

下一篇就讲这个。如果这一篇引起了共鸣，下一篇会把它落到能跑起来的状态。
